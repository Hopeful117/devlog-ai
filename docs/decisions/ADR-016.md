# ADR-016 - Separation Between Knowledge Events, Facts and Observations

## Status

Accepted

## Date

2026-07-21

## Context

DevLog AI analyzes the evolution and state of software projects.

The system receives or collects information from multiple sources such as:

* Git repositories;
* commits;
* branches;
* pull requests;
* dependency changes;
* file changes;
* project metadata;
* CI/CD systems;
* documentation;
* milestones;
* other project-related sources.

The system needs to distinguish between different levels of knowledge produced during analysis.

Without a clear separation, raw project events, objectively established information, deterministic conclusions, and AI interpretations could become mixed together.

This would make it difficult to:

* maintain traceability;
* identify the origin of information;
* distinguish objective facts from interpretations;
* determine which information can be trusted automatically;
* validate AI-generated conclusions;
* reproduce analyses;
* provide reliable context to the AI Engine.

The architecture therefore requires an explicit distinction between:

1. `KnowledgeEvent`;
2. `Fact`;
3. `Observation`;
4. AI interpretation;
5. `ValidatableProposal`;
6. validated project knowledge.

## Decision

DevLog AI will use a layered knowledge pipeline where each level has a clearly defined responsibility.

The conceptual pipeline is:

```text
Project Sources
      │
      ▼
KnowledgeEvent
      │
      ▼
Fact
      │
      ▼
Observation
      │
      ▼
AI Interpretation
      │
      ▼
ValidatableProposal
      │
      ▼
Human Validation
      │
      ▼
Validated Project Knowledge
```

The architectural principle is:

> Lower layers describe what happened or what can be objectively established. Higher layers interpret or derive meaning from those lower layers.

Each layer must remain distinguishable and traceable to its source.

---

## 1. KnowledgeEvent

A `KnowledgeEvent` represents an event that occurred in or around the project.

It answers:

> **What happened?**

A KnowledgeEvent is associated with a source and represents an occurrence that can be observed or imported by the system.

Examples include:

* a Git commit was created;
* a branch was merged;
* a pull request was opened;
* a dependency was added;
* a dependency version changed;
* a file was modified;
* a release was created;
* a project milestone was completed.

Conceptually:

```text
KnowledgeEvent
├── Source
├── Event Type
├── Timestamp
├── Project
└── Event Data
```

KnowledgeEvents represent the input material from which deterministic analysis can derive higher-level knowledge.

A KnowledgeEvent does not itself represent an interpretation of the project.

---

## 2. Fact

A `Fact` represents information that can be objectively established from project sources or deterministic processing.

It answers:

> **What can we objectively establish?**

A Fact must be based on identifiable evidence.

Examples include:

```text
The project uses Spring Boot.

The project contains a REST endpoint.

A dependency changed from version X to version Y.

The project contains no integration tests.

A class was moved from package A to package B.

The project contains a Docker Compose configuration.

The project uses PostgreSQL.
```

A Fact must not express subjective interpretation or speculation.

For example:

```text
Valid Fact:
"The project uses asynchronous REST communication."

Invalid Fact:
"The team introduced asynchronous REST communication
because they wanted better scalability."
```

The second statement contains an interpretation of intent and therefore does not qualify as a deterministic Fact.

Facts should be traceable to their underlying evidence.

Conceptually:

```text
Fact
├── Analysis
├── Fact Type
├── Value / Structured Data
├── Source References
└── Created At
```

The exact persistence model may evolve, but traceability must be preserved.

---

## 3. Observation

An `Observation` represents a deterministic conclusion derived from one or more Facts.

It answers:

> **What deterministic pattern can we identify?**

Observations are not raw events and are not simple facts.

They represent a higher-level pattern or relationship that can be established through deterministic analysis.

Examples include:

```text
The project is transitioning from synchronous
to asynchronous communication.

Multiple services now communicate through HTTP.

The architecture is becoming more modular.

The project introduced a new authentication layer.

Test coverage decreased after a specific change.
```

An Observation must be reproducible from its supporting Facts.

Conceptually:

```text
Fact A ─────┐
            │
Fact B ─────┼──► Deterministic Analysis ──► Observation
            │
Fact C ─────┘
```

An Observation must therefore maintain references to the Facts that support it.

An Observation must not introduce unsupported assumptions about:

* developer intent;
* business motivation;
* future consequences;
* subjective quality;
* undocumented reasoning.

For example:

```text
Valid Observation:
"Communication between services is now asynchronous."

Invalid Observation:
"The team introduced asynchronous communication
to improve scalability."
```

The second statement contains an assumption about intent and belongs to the AI interpretation layer unless explicit evidence exists.

---

## 4. AI Interpretation

The AI Engine operates above the deterministic knowledge layers.

It receives structured context containing relevant:

* Facts;
* Observations;
* KnowledgeEvents when necessary;
* project metadata;
* historical knowledge;
* supporting evidence.

The AI Engine answers questions such as:

> **What might this mean?**

Examples:

```text
Observation:
The project introduced asynchronous communication.

AI Interpretation:
This change may have been motivated by service decoupling,
scalability requirements, or long-running processing needs.
```

AI interpretations are inherently probabilistic and must therefore not be treated as trusted project knowledge automatically.

The AI Engine must not modify Facts or Observations directly.

The AI Engine produces proposals based on the context provided by the Core.

---

## 5. ValidatableProposal

AI-generated interpretations or derived knowledge must cross a validation boundary before becoming trusted project knowledge.

The `ValidatableProposal` represents the AI Engine's proposed contribution to the project's knowledge base.

The workflow is:

```text
Facts
   │
   ▼
Observations
   │
   ▼
AnalysisContext
   │
   ▼
AI Engine
   │
   ▼
ValidatableProposal
   │
   ▼
Human Validation
```

A ValidatableProposal may contain proposed:

* Insights;
* Decisions;
* Events;
* Documentation;
* other supported project knowledge.

The proposal must retain traceability to:

* the Project;
* the Analysis;
* the AI Task that produced it, when applicable;
* the supporting context or source references where applicable.

The exact structure is governed by ADR-006.

---

## 6. Validation Boundary

The trust model is explicitly divided into two categories.

### Trusted deterministic knowledge

The Core is considered authoritative for:

* KnowledgeEvents;
* Facts;
* Observations;
* deterministic analysis results.

These elements are produced or verified through deterministic mechanisms and are therefore trusted by the workflow.

The user does not need to manually validate every deterministic result.

### Untrusted AI-generated knowledge

AI-generated interpretations and derived knowledge are not automatically trusted.

They must pass through the `ValidatableProposal` boundary.

The user is responsible for validating the AI-generated proposal before it becomes trusted project knowledge.

The principle is:

```text
Deterministic Core Output
        │
        ▼
      Trusted
        │
        ▼
AI Interpretation
        │
        ▼
ValidatableProposal
        │
        ▼
 Human Validation
        │
        ▼
      Trusted
```

The system therefore does not challenge the deterministic Core output by default.

Instead, human vigilance is focused on the AI-generated layer.

---

## 7. Traceability

Traceability is a fundamental requirement of this architecture.

The system should allow the origin of an AI-generated proposal to be traced through the analysis pipeline.

Conceptually:

```text
ValidatableProposal
        │
        ▼
AI Task
        │
        ▼
Analysis
        │
        ▼
Observation
        │
        ▼
Fact
        │
        ▼
KnowledgeEvent
        │
        ▼
Project Source
```

The exact level of traceability may vary depending on the type of information.

However, the architecture must preserve sufficient relationships to explain:

* where a Fact came from;
* which Facts support an Observation;
* which Analysis produced the context;
* which AI Task generated a proposal;
* which human validation accepted or rejected it.

This enables:

* auditing;
* debugging;
* reproducibility;
* AI output evaluation;
* historical analysis;
* user trust.

---

## 8. Immutability and Historical Integrity

KnowledgeEvents, Facts, and Observations represent historical analysis data.

The system should avoid silently rewriting historical results.

When an analysis is rerun and produces different results, the system should preserve the distinction between analysis executions rather than silently replacing previous results.

The exact lifecycle and versioning strategy may evolve.

However, the following principle applies:

> Historical analysis results must remain traceable to the Analysis that produced them.

An `Analysis` therefore acts as the logical boundary for the Facts and Observations produced during a deterministic analysis execution.

---

## 9. Responsibility Boundaries

The responsibilities are divided as follows.

### KnowledgeEvent

Responsible for:

* representing project events;
* preserving source information;
* providing input to deterministic analysis.

### Fact

Responsible for:

* representing objectively established information;
* preserving evidence references;
* providing reliable input to deterministic reasoning.

### Observation

Responsible for:

* representing deterministic patterns;
* referencing supporting Facts;
* providing structured interpretation context.

### Core

Responsible for:

* collecting KnowledgeEvents;
* performing deterministic analysis;
* producing Facts;
* producing Observations;
* building AnalysisContext;
* orchestrating AI Tasks;
* persisting workflow state.

### AI Engine

Responsible for:

* interpreting Facts and Observations;
* generating hypotheses;
* identifying potential meaning;
* generating proposals.

The AI Engine does not own:

* project knowledge persistence;
* deterministic retrieval;
* Facts;
* Observations;
* the authoritative project state.

### User

Responsible for:

* reviewing AI-generated proposals;
* accepting or rejecting proposed interpretations;
* deciding which AI-generated knowledge becomes trusted project knowledge.

## 10. Alternatives Considered

### Treat KnowledgeEvents and Facts as the same concept

Rejected because:

* events describe occurrences;
* facts describe objectively established information;
* one event may produce multiple facts;
* facts may be derived from multiple events;
* separating the two improves traceability.

### Use only Facts and eliminate Observations

Rejected because:

* deterministic patterns would have no explicit representation;
* the distinction between objective information and deterministic conclusions would be lost;
* AI context construction would become less structured.

### Let the AI Engine generate Facts and Observations

Rejected because:

* Facts and Observations are deterministic outputs;
* allowing AI generation would blur the trust boundary;
* AI-generated information would require validation;
* the Core should remain responsible for deterministic analysis.

### Require human validation for Facts and Observations

Rejected because:

* deterministic Core outputs are considered trusted;
* manually validating every deterministic result would introduce unnecessary friction;
* the primary validation boundary is intended for AI-generated knowledge.

### Store only the final AI-generated knowledge

Rejected because:

* intermediate reasoning context would be lost;
* traceability would be reduced;
* debugging and reproducibility would become difficult;
* the system would not be able to explain how a proposal was produced.

---

## 11. Consequences

### Positive

* Clear separation between events, facts, observations, and interpretations.
* Strong traceability across the analysis pipeline.
* Clear trust boundary between deterministic and AI-generated information.
* Reduced need for human validation of deterministic results.
* Focuses human attention on the layer where uncertainty is highest.
* Improves AI context quality.
* Supports reproducible analysis.
* Makes the architecture easier to reason about.
* Provides clear responsibilities for Core and AI Engine.
* Allows future AI models to be replaced without changing deterministic knowledge production.

### Negative

* Introduces additional domain concepts.
* Requires relationships between Facts and supporting evidence.
* Requires relationships between Observations and supporting Facts.
* Increases persistence and workflow complexity.
* Requires careful handling of historical analysis results.
* Deterministic analysis logic may become complex as the number of supported Fact and Observation types grows.

---

## 12. Architectural Principle

DevLog AI follows the following knowledge hierarchy:

```text
KnowledgeEvent
"What happened?"
        │
        ▼
Fact
"What can we objectively establish?"
        │
        ▼
Observation
"What deterministic pattern can we identify?"
        │
        ▼
AI Interpretation
"What might this mean?"
        │
        ▼
ValidatableProposal
"What should become trusted knowledge?"
        │
        ▼
Human Validation
"Is this interpretation correct?"
        │
        ▼
Validated Project Knowledge
```

The fundamental rule is:

> **The Core determines what is objectively known. The AI Engine interprets what that knowledge might mean. The user validates AI-generated interpretations before they become trusted project knowledge.**

This separation is the foundation of DevLog AI's deterministic analysis and AI-assisted knowledge generation architecture.
