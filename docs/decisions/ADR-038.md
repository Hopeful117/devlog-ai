# ADR-038 — Repository Context Engine

* **Status:** Accepted
* **Date:** 2026-07-23

## Context

ADR-037 establishes the **Repository-First** strategy.

The repository is considered the primary source of project knowledge, and every AI analysis should maximize the use of repository-derived information before introducing external context.

However, repository information is heterogeneous and potentially very large.

A typical repository may contain:

* thousands of source files;
* long Git histories;
* numerous ADRs;
* roadmap documents;
* validated Insights;
* tests;
* generated documentation;
* configuration files.

Sending the complete repository to a Large Language Model is neither technically feasible nor desirable.

DevLog therefore requires a dedicated component responsible for constructing an optimized and explainable analysis context.

---

## Decision

DevLog introduces a **Repository Context Engine**.

The Context Engine is responsible for transforming repository knowledge into a bounded, structured and explainable `AnalysisContext` suitable for AI interpretation.

The Context Engine is deterministic.

Artificial Intelligence consumes the resulting context but does not participate in its construction.

---

## Responsibilities

The Context Engine is responsible for:

* discovering repository evidence;
* collecting contextual information;
* ranking repository evidence;
* selecting the most relevant information;
* enforcing token budgets;
* constructing the final `AnalysisContext`;
* preserving evidence traceability.

It does not perform semantic interpretation.

Interpretation remains the responsibility of the AI Engine.

---

## Architecture

The Context Engine follows the pipeline:

```text
Repository

        │

        ▼

Context Collectors

        │

        ▼

Evidence Ranking

        │

        ▼

Evidence Selection

        │

        ▼

Token Budget

        │

        ▼

AnalysisContext

        │

        ▼

Intent Execution

        │

        ▼

AI Engine
```

Each stage has a single responsibility.

---

## Context Collectors

Repository knowledge is collected through specialized collectors.

Typical collectors include:

* Source Code Collector;
* Git History Collector;
* Commit Diff Collector;
* ADR Collector;
* Roadmap Collector;
* Documentation Collector;
* Insight Collector;
* Test Collector;
* Configuration Collector.

Collectors expose a common interface while remaining independent.

New collectors may be added without modifying existing ones.

---

## Evidence Model

Every collected element becomes an **Evidence** object.

Evidence should retain:

* source type;
* repository location;
* originating file;
* identifier;
* timestamp when applicable;
* relevance metadata;
* extraction metadata.

Evidence remains immutable.

---

## Evidence Ranking

Collected evidence is ranked according to the current analysis.

Ranking criteria may include:

* Intent;
* User Guidance;
* semantic proximity;
* architectural relationships;
* Git history;
* roadmap relationships;
* ADR references;
* dependency graph;
* repository location;
* recency;
* validated knowledge.

Ranking algorithms may evolve independently from collectors.

---

## Evidence Selection

Only the highest-value evidence is selected.

Selection should maximize:

* usefulness;
* diversity;
* explainability;
* traceability.

Redundant information should be avoided whenever possible.

---

## Context Profiles

Different analyses require different repository perspectives.

The Context Engine therefore supports **Context Profiles**.

Examples include:

* Project State;
* Architecture Review;
* Documentation;
* README Generation;
* Release Summary;
* Knowledge Extraction;
* History Analysis.

Profiles influence collector priorities and ranking strategies.

---

## Token Budget Management

The Context Engine must operate within configurable token limits.

Strategies may include:

* evidence ranking;
* summarization;
* chunking;
* duplicate elimination;
* context compression.

The objective is to maximize information quality rather than token quantity.

---

## Explainability

Every AI analysis must remain explainable.

The resulting `AnalysisContext` should preserve references to the evidence used.

Future versions should allow users to inspect:

* selected evidence;
* discarded evidence;
* ranking decisions;
* token allocation.

---

## Extensibility

The Context Engine is repository-centric.

Future versions may integrate additional context providers, including:

* issue trackers;
* pull requests;
* CI/CD pipelines;
* monitoring systems;
* architecture repositories;
* external documentation.

These providers participate through the same collector abstraction.

Repository evidence remains the preferred source.

---

## Design Principles

The Context Engine should remain:

* deterministic;
* modular;
* explainable;
* provider-independent;
* extensible;
* traceable.

The AI Engine should consume structured knowledge rather than raw repository data.

---

## Consequences

### Advantages

* higher-quality AI prompts;
* reduced hallucinations;
* deterministic context construction;
* reusable repository evidence;
* improved explainability;
* scalable architecture.

### Trade-offs

* additional preprocessing;
* ranking complexity;
* token-budget management;
* collector orchestration.

These trade-offs are accepted because context quality directly determines analysis quality.

---

## Alternatives Considered

### Send Entire Repository to the LLM

Rejected.

Large repositories exceed practical token limits and introduce excessive noise.

---

### Context Construction Inside the AI Engine

Rejected.

Context construction should remain deterministic.

The AI Engine should interpret context rather than build it.

---

### Single Monolithic Context Builder

Rejected.

Specialized collectors and ranking strategies provide greater extensibility and maintainability.
