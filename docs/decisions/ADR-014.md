# ADR-014 - Hybrid Context Retrieval Owned by the Core

## Status

Accepted

## Date

2026-07-21

## Context

DevLog AI needs to construct relevant context for AI tasks.

The deterministic Core owns the project's structured knowledge and is responsible for preparing the context sent to the AI Engine.

As the project grows, retrieving relevant knowledge may require more than structured database queries.

Deterministic retrieval is effective when relationships and metadata are explicit.

Examples include:

* project identifiers,
* entity types,
* dates,
* technologies,
* components,
* related analyses,
* explicit entity relationships.

However, semantic relationships may not always be explicitly represented.

For example, a user may ask:

> "Why did we move toward asynchronous architecture?"

Relevant knowledge may refer to:

* Kafka,
* message brokers,
* decoupling,
* asynchronous processing,
* event-driven architecture.

The relevant concepts may be semantically related without sharing the same exact keywords or explicit relationships.

DevLog AI may therefore benefit from combining deterministic retrieval with semantic retrieval.

## Decision

DevLog AI will use a hybrid retrieval strategy when semantic retrieval is introduced.

The Java Core will remain responsible for the complete retrieval process.

The retrieval layer may combine:

1. deterministic retrieval based on structured project knowledge;
2. semantic retrieval based on conceptual similarity.

The resulting information will be combined and used to construct the `AnalysisContext` sent to the AI Engine.

The conceptual architecture is:

```text id="m7q2v9"
                     AI Task
                        │
                        ▼
                Core Retrieval Layer
                        │
              ┌─────────┴─────────┐
              │                   │
              ▼                   ▼
       Deterministic          Semantic
        Retrieval             Retrieval
              │                   │
              │              Vector Search
              │                   │
              └─────────┬─────────┘
                        ▼
                Context Selection
                        │
                        ▼
                 Context Ranking
                        │
                        ▼
                AnalysisContext
                        │
                        ▼
                   AI Engine
```

The semantic retrieval mechanism is considered a retrieval capability of the Core architecture rather than a responsibility of the AI Engine.

## Deterministic Retrieval

Deterministic retrieval will remain the primary mechanism for information that can be identified through explicit project relationships and structured metadata.

Examples include:

```text id="q9x3m4"
Project = X
Technology = Java
Component = Authentication
Date > 2026-01-01
Analysis = Y
```

This type of retrieval provides predictable and traceable results.

## Semantic Retrieval

Semantic retrieval may be introduced to identify conceptually related information.

For example:

```text id="k3p7z1"
Query:
"Why did we introduce asynchronous processing?"

Potential semantic matches:
- Kafka adoption
- Message broker integration
- Service decoupling
- Event-driven architecture
- Asynchronous job processing
```

Semantic retrieval should complement deterministic retrieval rather than replace it.

## Context Assembly

The Core is responsible for combining retrieved information into a structured `AnalysisContext`.

Conceptually:

```text id="r8v2n5"
Deterministic Results
        │
        ├──────────────┐
        │              │
        ▼              ▼
Semantic Results   Explicit Relations
        │              │
        └──────┬───────┘
               ▼
        Deduplication
               │
               ▼
        Relevance Ranking
               │
               ▼
        Context Assembly
               │
               ▼
        AnalysisContext
```

The exact ranking and deduplication strategy may evolve as semantic retrieval is implemented.

## Ownership Principle

The following architectural rule applies:

> The AI Engine interprets the context. The Core determines which knowledge belongs in the context.

The AI Engine must not directly query the Core's knowledge store to perform its own retrieval.

The Core remains responsible for:

* retrieval,
* filtering,
* ranking,
* deduplication,
* context assembly.

The AI Engine remains responsible for:

* interpretation,
* reasoning,
* hypothesis generation,
* proposal generation.

## Alternatives Considered

### AI Engine owns semantic retrieval

The Python AI Engine would maintain embeddings and perform vector searches against project knowledge.

Rejected because:

* knowledge retrieval would become coupled to AI infrastructure,
* Python would maintain its own representation of project knowledge,
* synchronization between Core knowledge and AI retrieval indexes would be required,
* the Core would lose control over context selection.

### Semantic retrieval only

Use vector search as the sole mechanism for retrieving project knowledge.

Rejected because:

* explicit relationships are often more reliable than semantic similarity,
* deterministic queries provide better traceability,
* structured filtering is more precise for dates, types, projects, and components,
* semantic retrieval can return conceptually similar but contextually incorrect information.

### Deterministic retrieval only

Use only structured queries indefinitely.

Rejected as a long-term strategy because:

* semantic relationships may not be explicitly modeled,
* relevant historical knowledge may use different terminology,
* large knowledge bases become harder to navigate using only predefined relations.

This remains the strategy for V1 and is formalized separately in ADR-015.

### Dedicated retrieval microservice

Create a separate service responsible for deterministic and semantic retrieval.

Rejected because:

* it adds architectural complexity,
* retrieval remains tightly connected to the Core's knowledge model,
* a dedicated service is not justified by current V1 requirements.

This option may be reconsidered if retrieval becomes a sufficiently independent and resource-intensive capability.

## Consequences

### Positive

* Combines the precision of deterministic retrieval with the flexibility of semantic search.
* Keeps knowledge ownership and retrieval responsibility within the Core.
* Improves the quality of context provided to AI.
* Allows semantic retrieval to be introduced incrementally.
* Avoids coupling the AI Engine to the project's persistence model.
* Preserves the possibility of replacing AI models independently of retrieval infrastructure.

### Negative

* Hybrid retrieval is more complex than deterministic retrieval alone.
* Semantic retrieval introduces additional infrastructure.
* Ranking and deduplication strategies must be designed.
* Semantic search may return false-positive or contextually irrelevant results.
* The Core will eventually need to manage embedding and vector-search infrastructure.
