# ADR-027 — Immutable Deterministic Project Profile Snapshot

## Status

Accepted

## Date

2026-07-22

---

## Context

ADR-023 introduced the deterministic knowledge collection pipeline.

ADR-024 enriched repository analysis with specialized Collectors.

ADR-026 defined Observations as deterministic, reproducible and traceable characteristics derived from Facts.

DevLog AI can now identify objective characteristics such as:

- a project being containerized;
- a Spring Boot REST application being present;
- ADR documentation being available;
- automated and integration tests being detected;
- a multi-module build being used.

These Observations are useful individually, but they do not yet form a stable, structured representation of the analyzed project.

A future frontend, the AI Engine, historical comparison, generated documentation and the planned Engineering Hub will require a common contract answering:

> What objectively characterizes this project at this specific revision?

Generating a flat text description directly from Observations would be simple, but it would make human-readable text the primary contract. That approach would make structured consumption, historical comparisons and future extensions difficult.

DevLog AI therefore requires an immutable and versioned project profile derived deterministically from persisted Observations.

---

## Decision

DevLog AI introduces `ProjectProfileSnapshot`.

A `ProjectProfileSnapshot` is the immutable, structured and deterministic representation of a Project at the revision analyzed by a specific `Analysis`.

It is built exclusively from:

- persisted Observations belonging to the Analysis;
- Analysis metadata;
- the requested and resolved revisions;
- persisted collection diagnostics;
- stable profile-building rules.

It must never be built directly from:

- repository files;
- workspace contents;
- raw Facts without an Observation;
- AI-generated content;
- current project state reconstructed after the Analysis.

The workflow becomes:

```text
Workspace
    ↓
Collectors
    ↓
Facts
    ↓
Observation Engine
    ↓
Observations
    ↓
Project Profile Builder
    ↓
ProjectProfileSnapshot
    ↓
AnalysisContext Builder
    ↓
AI Engine
```

---

## Definition

A `ProjectProfileSnapshot` answers:

> What objective and deterministic characteristics described this Project when this Analysis was executed?

It does not answer:

- whether the architecture is good;
- whether documentation is sufficient;
- whether a technology should be replaced;
- what the developers intended;
- what the project should become.

Those interpretations belong to AI-generated proposals and human validation.

---

## Cardinality

The relationship is:

```text
Analysis 1 ───── 0..1 ProjectProfileSnapshot
```

An Analysis may have no profile when it fails before profile generation.

A successfully created profile becomes immutable.

A profile is never updated after creation.

A new Analysis produces a new snapshot.

---

## Persistence

The snapshot is persisted as historical project knowledge derived from the Analysis.

Persistence guarantees:

- historical reproducibility;
- independence from future Observation rule changes;
- exact exposure of the profile produced at analysis time;
- reliable comparison between two revisions;
- reuse by the API, frontend and AI Engine.

Older profiles must never be reconstructed dynamically using newer rules.

---

## Core Model

Conceptually:

```text
ProjectProfileSnapshot
├── id
├── projectId
├── analysisId
├── profileVersion
├── rendererVersion
├── generatedAt
├── requestedRevision
├── resolvedRevision
├── completeness
├── sections
├── deterministicSummary
└── sourceObservationIds
```

The exact Java implementation may use normalized relational entities, immutable JSON snapshots or a combination of both, provided that immutability and traceability are preserved.

---

## Profile Version

Every profile uses an explicit semantic version identifier.

Initial version:

```text
project-profile-v1
```

A new profile version is required when changes materially affect:

- section taxonomy;
- characteristic semantics;
- profile-building rules;
- completeness interpretation;
- serialized contract.

Minor implementation refactoring that does not change output semantics does not require a new profile version.

---

## Section Taxonomy

The profile uses a stable high-level taxonomy.

Initial categories are:

```text
IDENTITY
TECHNOLOGIES
ARCHITECTURE
BUILD
API
PERSISTENCE
SECURITY
TESTING
DOCUMENTATION
INFRASTRUCTURE
CI_CD
QUALITY
```

A section may be empty when no corresponding Observation exists.

The absence of a characteristic must not be interpreted as proof that the characteristic is absent unless a dedicated deterministic rule establishes that absence.

For example:

```text
No SECURITY characteristic
```

does not mean:

```text
The project has no security.
```

It means only that the current deterministic analysis produced no supported security characteristic.

---

## Profile Section

Conceptually:

```text
ProfileSection
├── category
├── characteristics
└── deterministicSummary
```

Sections are ordered deterministically according to the profile taxonomy.

Empty sections may be omitted from API responses while remaining conceptually part of the profile contract.

---

## Profile Characteristic

Each profile characteristic represents one objective project property supported by one or more Observations.

Conceptually:

```text
ProfileCharacteristic
├── code
├── label
├── description
├── status
├── sourceObservationIds
├── evidenceCount
└── metadata
```

Initial examples include:

```text
SPRING_BOOT_REST_APPLICATION
CONTAINERIZED_PROJECT
ADR_DOCUMENTATION
AUTOMATED_TESTS
INTEGRATION_TESTS
MULTI_MODULE_BUILD
```

The characteristic `code` is the stable machine-readable identity.

Labels and descriptions are presentation data and may evolve independently through renderer versioning.

---

## Characteristic Status

The profile must not use probabilistic confidence scores.

Scores such as:

```text
0.84
```

would create false precision for deterministic results.

Initial supported statuses are:

```text
CONFIRMED
PARTIALLY_CONFIRMED
```

`CONFIRMED` means that all deterministic conditions of the corresponding profile rule are satisfied.

`PARTIALLY_CONFIRMED` may only be used when an explicit profile rule defines a meaningful deterministic partial state.

A characteristic must not use `PARTIALLY_CONFIRMED` merely because collection warnings exist.

Collection completeness remains separate metadata.

---

## Traceability

Every characteristic must reference all Observations that justify it.

The complete traceability chain is:

```text
ProjectProfileCharacteristic
    ↓
Observation
    ↓
ObservationRule
    ↓
Fact
    ↓
Evidence
```

A profile characteristic without at least one persisted Observation is invalid.

Observations from another Analysis or Project must be rejected.

---

## Source Observation Snapshot

The profile stores the identifiers of all Observations used to build it.

The snapshot must preserve:

- Observation ID;
- Observation type;
- rule ID;
- rule version.

This metadata may be embedded in characteristics or stored as dedicated profile traceability records.

---

## Completeness

The profile exposes the quality and completeness of the deterministic collection from which it was derived.

Conceptually:

```text
ProfileCompleteness
├── status
├── collectionComplete
├── truncated
├── warningCount
├── errorCount
├── successfulCollectorCount
├── collectorWithWarningsCount
└── failedCollectorCount
```

Initial status values:

```text
COMPLETE
PARTIAL
FAILED
```

### COMPLETE

Used when:

- deterministic collection completed;
- no collector failed;
- no configured limit caused truncation;
- no significant warning indicates incomplete evidence.

### PARTIAL

Used when:

- the Analysis completed;
- usable Facts and Observations exist;
- but warnings, truncation or non-fatal collector failures indicate incomplete coverage.

### FAILED

Used only when profile generation is explicitly retained for a failed or unusable collection.

For V1, an Analysis that fails before profile generation should normally have no snapshot rather than a `FAILED` profile.

---

## Deterministic Profile Builder

A dedicated component builds the snapshot.

Conceptually:

```text
ProjectProfileBuilder
```

Responsibilities:

1. load the Analysis and its persisted diagnostics;
2. load all persisted Observations belonging to the Analysis;
3. validate Observation ownership and traceability;
4. map Observations to profile sections;
5. build deterministic characteristics;
6. calculate completeness;
7. order sections and characteristics stably;
8. generate the deterministic summary;
9. persist the snapshot atomically.

The builder must never:

- access the Git workspace;
- execute Collectors;
- query the AI Engine;
- infer subjective characteristics;
- inspect current Project state unrelated to the Analysis;
- silently ignore unsupported Observation types.

Unsupported Observation types should generate an explicit diagnostic or fail according to a documented mapping policy.

---

## Observation-to-Profile Mapping

Observation types map deterministically to profile categories.

Initial mapping:

```text
SPRING_BOOT_REST_APPLICATION
→ ARCHITECTURE
→ TECHNOLOGIES
→ API

CONTAINERIZED_PROJECT
→ INFRASTRUCTURE

ADR_DOCUMENTATION
→ DOCUMENTATION

AUTOMATED_TESTS
→ TESTING

INTEGRATION_TESTS
→ TESTING

MULTI_MODULE_BUILD
→ BUILD
→ ARCHITECTURE
```

A single Observation may support more than one characteristic when the mappings represent distinct objective dimensions.

Duplicate characteristics must be prevented using a stable deterministic identity.

---

## Stable Ordering

Profile output must remain independent of:

- database retrieval order;
- Observation insertion order;
- Collector execution timing.

Sections are ordered by the fixed taxonomy.

Characteristics are ordered by:

1. category;
2. characteristic code;
3. stable source Observation identity when necessary.

The same Observations and profile version must produce the same serialized profile.

---

## Deterministic Summary

The profile includes a human-readable deterministic summary.

The summary is derived exclusively from the structured profile.

It is never the source of truth.

Conceptually:

```text
ProjectProfileSnapshot
    ↓
ProjectProfileRenderer
    ↓
Human-readable summary
```

Initial renderer:

```text
project-profile-renderer-fr-v1
```

The renderer must be:

- deterministic;
- versioned;
- independent from any LLM;
- based only on profile sections and characteristics;
- safe when the profile is partial.

Example:

> Le projet est une application REST développée avec Spring Boot. Il est conteneurisé, documenté avec des ADR et comporte des tests automatisés, dont des tests d’intégration.

For a partial profile, the renderer must explicitly avoid suggesting exhaustiveness.

Example:

> L’analyse partielle a confirmé que le projet est conteneurisé et utilise une application REST Spring Boot. Certaines caractéristiques peuvent ne pas avoir été détectées.

---

## Localization

French is the initial rendering language.

The structured snapshot remains language-independent.

Future renderers may include:

```text
project-profile-renderer-en-v1
```

Adding a renderer must not modify the underlying snapshot.

---

## Immutability

After persistence, the following elements cannot change:

- profile version;
- Analysis association;
- revisions;
- completeness;
- sections;
- characteristics;
- source Observation references;
- deterministic summary;
- renderer version;
- generated timestamp.

Corrections or new semantics require:

- a new Analysis;
- a new snapshot;
- or a new profile version generated through an explicit migration process that does not overwrite historical data.

---

## Transaction Boundary

The following operations must occur atomically:

- profile validation;
- characteristic creation;
- Observation traceability creation;
- deterministic summary creation;
- snapshot persistence.

Either the complete profile is stored or no profile is stored.

---

## Analysis Workflow Integration

Profile generation occurs after Observation persistence and before `AnalysisContext` construction.

```text
Persist Facts
    ↓
Persist Observations
    ↓
Build and persist ProjectProfileSnapshot
    ↓
Build AnalysisContext
    ↓
Submit AI Task
```

If profile generation fails:

```text
Analysis → FAILED
```

The AI task must not be submitted with a missing or inconsistent profile when the profile is required by the active `AnalysisContext` version.

---

## AnalysisContext Integration

The persisted profile becomes part of the immutable `AnalysisContext`.

The AI Engine receives:

- selected Facts;
- selected Observations;
- the structured project profile;
- profile completeness;
- validated project knowledge;
- analysis metadata.

The context must reference:

```text
profileId
profileVersion
```

and embed a snapshot suitable for the AI contract.

The AI Engine must not rebuild or reinterpret the profile from raw Observations.

---

## API

The historical profile is exposed through:

```http
GET /api/v1/analyses/{analysisId}/profile
```

The endpoint returns the persisted immutable snapshot.

It never rebuilds the profile.

A missing profile returns an appropriate not-found response.

A convenience endpoint may also be implemented:

```http
GET /api/v1/projects/{projectId}/latest-profile
```

This endpoint returns the most recent persisted profile according to a deterministic ordering.

It must never trigger profile generation.

---

## Diagnostics Integration

`AnalysisDiagnostics` should expose a lightweight profile summary.

Conceptually:

```text
profileAvailable
profileId
profileVersion
profileCompleteness
characteristicCount
```

The complete profile remains available through its dedicated endpoint.

---

## Future Comparison

The snapshot contract must support future deterministic comparison.

Conceptually:

```text
ProjectProfileSnapshot A
    ↓
ProjectProfileComparison
    ↑
ProjectProfileSnapshot B
```

Future comparison may expose:

- added characteristics;
- removed characteristics;
- unchanged characteristics;
- changed completeness;
- changed revision;
- changed profile version.

Comparison implementation is outside the scope of this ADR.

---

## Security and Privacy

The profile must not contain:

- absolute workspace paths;
- repository credentials;
- API keys;
- raw secret values;
- full source files;
- raw environment configuration;
- unredacted sensitive metadata.

Characteristics must use normalized technical descriptions and relative evidence references through their Observation traceability.

---

## Initial Implementation Scope

This increment includes:

- `ProjectProfileSnapshot` persistence;
- profile sections and characteristics;
- completeness metadata;
- Observation traceability;
- deterministic builder;
- French deterministic renderer;
- profile and renderer versioning;
- workflow integration before `AnalysisContext`;
- profile inclusion in `AnalysisContext`;
- `GET /api/v1/analyses/{analysisId}/profile`;
- optional latest Project profile endpoint if consistent with existing conventions;
- diagnostics profile summary;
- Flyway migration;
- unit tests;
- integration tests;
- end-to-end validation on a real repository;
- updated architecture and API documentation.

This increment excludes:

- AI-generated project descriptions;
- editable profiles;
- manually entered characteristics;
- profile comparison endpoint;
- user-defined taxonomies;
- multiple renderer languages beyond French;
- profile reconstruction for historical Analyses;
- subjective quality assessments;
- recommendations;
- profile generation from Facts without Observations.

---

## Alternatives Considered

### Render text directly from Observations

Rejected because:

- text would become the effective contract;
- structured API consumption would be limited;
- historical comparison would require parsing human-readable text;
- frontend and external integrations would lack a stable model.

### Compute the profile dynamically

Rejected because:

- future rule changes would alter historical results;
- reproducibility would be lost;
- the exposed profile might differ from the one used by the AI Engine.

### Let the AI generate the project profile

Rejected because:

- the profile must remain deterministic;
- AI output is probabilistic;
- traceability and reproducibility would be weakened;
- AI-generated interpretations belong to `ValidatableProposal`.

### Store only profile codes without rendering

Rejected because:

- a deterministic human-readable description is useful immediately;
- the renderer can remain a separate, versioned projection;
- users should be able to understand the profile without an LLM.

---

## Consequences

### Positive

- Establishes a stable deterministic representation of a project.
- Preserves historical project state by revision.
- Creates a reusable contract for API, frontend and AI.
- Enables future profile comparisons.
- Improves AI context quality.
- Provides an immediate human-readable project description.
- Preserves complete traceability to Observations, Facts and Evidence.
- Avoids future extraction of structure from free-form text.
- Prepares integration with DevLog AI documentation, Codeglyphe and the Engineering Hub.

### Negative

- Adds a new persistent aggregate.
- Requires explicit taxonomy and mapping maintenance.
- Introduces profile and renderer versioning.
- Makes workflow execution dependent on profile generation.
- Requires careful migration when profile semantics evolve.
- May contain empty sections until the Observation ontology grows.

---

## Architectural Principle

DevLog AI distinguishes:

```text
Facts
→ What objectively exists.

Observations
→ What can be deterministically concluded.

Project Profile
→ How those conclusions describe the project as a structured whole.

AI Insights
→ What this structured description may mean.

Human Validation
→ What becomes trusted knowledge.
```

The fundamental rule is:

> **The ProjectProfileSnapshot is the immutable deterministic identity of a Project at a specific analyzed revision. Human-readable descriptions are versioned projections of that structured identity, never the source of truth.**
