## Prompt Versioning

Every Prompt Template is explicitly versioned.

Example:

```text
describe-project

v1
```

The Prompt Version represents the business identity of the template.

It is intended for humans and expresses stable prompting semantics.

Changing the Prompt Version requires a semantic modification capable of influencing the generated Insight Proposals.

Examples include:

- changing the role assigned to the model;
- modifying grounding rules;
- changing Human-in-the-Loop constraints;
- changing the expected output structure;
- changing deterministic context selection;
- modifying template composition;
- altering the interpretation strategy of an Intent.

Editorial changes that cannot reasonably influence model behavior do not require a new Prompt Version.

Examples include:

- spelling corrections;
- formatting improvements;
- comments;
- wording clarifications preserving identical semantics.

## Prompt Digest

In addition to the Prompt Version, every rendered Prompt produces a deterministic Prompt Digest.

Example:

```text
SHA-256(
    normalizedSystemMessage
    + normalizedUserMessage
    + normalizedExpectedSchema
)
```

The Prompt Digest uniquely identifies the exact prompt executed by the provider.

Unlike the Prompt Version, the digest changes whenever the rendered Prompt content changes.

Its purpose is:

- replay verification;
- diagnostics;
- accidental modification detection;
- execution traceability;
- reproducibility.

The Prompt Digest is a technical identifier.

It is not intended to replace the Prompt Version.

---

## Architectural Principle

Prompt Version expresses business intent.

Prompt Digest proves technical identity.

Both identifiers are complementary.

Prompt Version supports architectural governance.

Prompt Digest guarantees execution reproducibility.

Historical analyses must preserve both values.

No historical Prompt may be reconstructed using another Prompt Version while claiming to reproduce the original execution.
