# ADR-011 - Core Orchestration and AI Engine Processing Autonomy

## Status

Accepted

## Date

2026-07-21

## Context

DevLog AI separates deterministic project analysis from AI interpretation.

The Java Core owns the project's authoritative knowledge and the overall business workflow.

The Python AI Engine is responsible for executing AI processing tasks, which may involve multiple internal steps such as:

* context processing,
* prompt construction,
* model selection,
* model inference,
* response parsing,
* proposal generation.

The architecture must clearly define which service owns the overall workflow and which service owns the internal execution of an AI task.

Allowing the AI Engine to own the entire business workflow would make Python responsible for project knowledge and business state.

Conversely, requiring the Core to control every internal AI processing step would create unnecessary coupling between the two services and prevent the AI Engine from evolving independently.

## Decision

The Java Core will remain the orchestrator of the overall business workflow.

The Python AI Engine will be autonomous in the execution of individual AI processing tasks.

The responsibility boundary is:

```text id="1l4myj"
                         JAVA CORE
                   Business Orchestrator
                             │
                             │ AI Task Request
                             ▼
                     PYTHON AI ENGINE
                  AI Processing Engine
                             │
                ┌────────────┼────────────┐
                │            │            │
                ▼            ▼            ▼
          Context       Model        Inference
          Processing    Selection    Execution
                │            │            │
                └────────────┼────────────┘
                             ▼
                      AI Task Result
                             │
                             ▼
                         JAVA CORE
```

The Core decides **when** an AI task should be executed and **why** it should be executed.

The AI Engine decides **how** the AI task is internally executed.

## Java Core Responsibilities

The Core owns:

* the business workflow,
* `Analysis`,
* `Facts`,
* `Observations`,
* `AnalysisContext`,
* `AI Task` lifecycle,
* `ValidatableProposal`,
* validation workflows,
* trusted project knowledge.

The Core determines:

```text id="k4x7h2"
WHEN should AI processing happen?
WHY is AI processing required?
WHAT context should be provided?
WHAT should happen with the result?
```

## Python AI Engine Responsibilities

The AI Engine owns:

* AI task execution,
* model selection,
* prompt construction,
* internal AI processing steps,
* model inference,
* response parsing,
* generation of structured AI results.

The AI Engine determines:

```text id="n8p3v6"
HOW should the AI task be processed?
WHICH model should be used?
HOW should the prompt be constructed?
HOW should inference be performed?
```

The internal processing workflow of the AI Engine is considered an implementation detail of the AI service.

The Core must not depend on the internal steps used by Python to produce a result.

## Workflow

The resulting architecture is:

```text id="u5m2q9"
Java Core
    │
    ├── Analysis
    │
    ├── Facts
    │
    ├── Observations
    │
    ├── Context Retrieval
    │
    └── AnalysisContext
             │
             ▼
        AI Task Request
             │
             ▼
       Python AI Engine
             │
             ├── Process Context
             ├── Select Model
             ├── Build Prompt
             ├── Execute Inference
             └── Generate Result
             │
             ▼
        AI Task Result
             │
             ▼
          Java Core
             │
             ▼
    ValidatableProposal
             │
             ▼
      Human Validation
```

The AI Engine may internally perform multiple AI operations or use multiple models without requiring changes to the Core, as long as the external task contract remains stable.

## Alternatives Considered

### AI Engine owns the complete business workflow

The Python service would create analyses, manage project knowledge, control validation, and determine the lifecycle of AI-generated information.

Rejected because:

* it would make Python responsible for domain knowledge,
* business state would become distributed across services,
* the Core would lose its role as the authoritative system,
* AI implementation details would become coupled to business workflows.

### Core orchestrates every AI processing step

The Core would explicitly control:

```text id="k9w4v1"
Prompt creation
Model selection
Inference
Parsing
AI chaining
```

Rejected because:

* it tightly couples Java to AI implementation details,
* it prevents the AI Engine from evolving independently,
* it makes the Core aware of internal AI processing strategies,
* it increases cross-service communication.

### Shared workflow state between Core and AI Engine

Both services would maintain an authoritative representation of the AI workflow.

Rejected because:

* it creates competing sources of truth,
* synchronization becomes more complex,
* state conflicts become possible,
* failure recovery becomes harder to reason about.

The Core remains authoritative for business workflow state, while the AI Engine may maintain internal execution state.

## Consequences

### Positive

* Clear separation between business orchestration and AI execution.
* Core remains the authoritative system.
* AI implementation can evolve independently.
* Multiple AI models or processing strategies can be introduced without redesigning the Core workflow.
* Python remains focused on AI capabilities.
* The architecture minimizes the business responsibilities delegated to Python.

### Negative

* The Core must maintain the AI task lifecycle.
* The boundary between business state and internal AI state must remain explicit.
* Cross-service contracts must be carefully designed.
* Debugging asynchronous processing may require correlation identifiers and execution metadata.
