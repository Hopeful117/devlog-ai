# ADR-024 ‚Äî Initial Deterministic Collectors for Repository Analysis

## Status

Accepted

## Date

2026-07-22

## Context

ADR-023 introduced the deterministic knowledge collection pipeline:

```text
Workspace
    ‚Üì
Collectors
    ‚Üì
Facts
    ‚Üì
Observation Engine
    ‚Üì
Observations
    ‚Üì
AnalysisContext
    ‚Üì
AI Engine
```

The first implementation introduced a minimal `GitCollector` capable of producing deterministic commit-related Facts.

This validates the collection architecture, but the resulting `AnalysisContext` is not yet rich enough to describe the structure and technical characteristics of a real software project.

DevLog AI must now collect additional objective information from a repository while preserving the architectural principles established by ADR-016 and ADR-023:

- Collectors produce Facts only;
- Collectors remain deterministic and idempotent;
- Collectors do not infer developer intent;
- Collectors do not produce recommendations;
- the Observation Engine remains the only deterministic component allowed to derive patterns from several Facts;
- the AI Engine interprets only the resulting `AnalysisContext`.

The initial enrichment must remain sufficiently limited to avoid creating a large static-analysis framework before the first real end-to-end repository analysis.

## Decision

DevLog AI will introduce an initial bounded set of specialized deterministic Collectors.

The V1 collection pipeline will support:

```text
Workspace
    ‚îÇ

    ‚îú‚îÄ‚îÄ RepositoryMetadataCollector

    ‚îú‚îÄ‚îÄ DockerCollector
    ‚îú‚îÄ‚îÄ DocumentationCollector

```

Each Collector owns one explicit technical concern and produces typed Facts supported by direct repository evidence.

No new Collector may produce Observations, Insights, recommendations, or AI proposals.

## Common Collector Contract

All Collectors implement a common contract.

Conceptually:

```java
public interface KnowledgeCollector {

    boolean supports(CollectionContext context);

    CollectionResult collect(CollectionContext context);
}
```

The exact Java contract may differ according to existing project conventions.

The contract must preserve the following responsibilities:

- determine whether the Collector supports the current Source and workspace;
- inspect only the immutable synchronized workspace supplied by the collection workflow;
- return deterministic Fact candidates;
- return collection metadata and non-fatal warnings where appropriate;
- avoid direct business persistence.

The orchestration layer remains responsible for validating and persisting the resulting Facts.

## Collection Context

Collectors receive an immutable collection context containing only the information required for deterministic inspection.

Conceptually:

```text
CollectionContext
‚îú‚îÄ‚îÄ analysisId
‚îú‚îÄ‚îÄ sourceId
‚îú‚îÄ‚îÄ projectId
‚îú‚îÄ‚îÄ workspacePath
‚îú‚îÄ‚îÄ resolvedRevision
‚îú‚îÄ‚îÄ sourceType
‚îî‚îÄ‚îÄ collectionTimestamp
```

Collectors must not:

- load JPA entities independently;
- access repositories directly;
- modify the workspace;
- synchronize Git;
- call the AI Engine;
- access external provider APIs.

Workspace preparation and Git synchronization remain infrastructure responsibilities outside individual Collectors.

## Collection Result

Each Collector returns a structured result.

Conceptually:

```text
CollectionResult
‚îú‚îÄ‚îÄ collectorType
‚îú‚îÄ‚îÄ collectorVersion
‚îú‚îÄ‚îÄ facts
‚îú‚îÄ‚îÄ warnings

```

A Collector must not persist its result directly.

Fact persistence remains coordinated by the deterministic analysis workflow.

## Fact Requirements

Every generated Fact must contain enough information to explain its origin.

At minimum, a collected Fact should expose:

```text
Fact
‚îú‚îÄ‚îÄ type
‚îú‚îÄ‚îÄ content
‚îú‚îÄ‚îÄ source
‚îú‚îÄ‚îÄ evidenceReferences
‚îú‚îÄ‚îÄ detectedAt
‚îî‚îÄ‚îÄ collector metadata
```

Evidence references may include:

- relative file paths;
- Git commit identifiers;
- build descriptor paths;
- configuration file paths;
- source directories.

Absolute workspace paths must never be persisted because they are infrastructure-specific and may expose host details.

## Collector Versioning

Every Collector has an explicit version.

Initial versions use:

```text
repository-metadata-v1
build-v1
spring-v1
docker-v1
documentation-v1
test-structure-v1
```

Collector versions allow DevLog AI to determine which deterministic logic produced a Fact.

A future change that materially changes collection semantics must increment the Collector version.

## 1. RepositoryMetadataCollector

The `RepositoryMetadataCollector` describes the general structure of the synchronized repository.

It may produce Facts for:

- resolved Git revision;
- default or targeted branch;
- repository root file count;
- repository directory count;
- detected source directories;
- detected primary file extensions;
- presence of common configuration files;
- presence of a multi-module structure.

Examples:

```text
REPOSITORY_REVISION_RESOLVED
SOURCE_DIRECTORY_PRESENT
PRIMARY_FILE_EXTENSION
MULTI_MODULE_STRUCTURE_PRESENT
CONFIGURATION_FILE_PRESENT
```

The Collector must not infer:

- project quality;
- architectural maturity;
- team organization;
- repository complexity.

For example:

```text
Valid Fact:
"The repository contains three Maven module descriptors."

Invalid Fact:
"The repository has a well-designed modular architecture."
```

## 2. BuildCollector

The `BuildCollector` detects supported build systems and extracts objective build metadata.

The initial V1 supports:

- Maven;
- Gradle.

It may inspect:

```text
pom.xml
settings.xml
build.gradle
build.gradle.kts
settings.gradle
settings.gradle.kts
gradle.properties
mvnw
gradlew
```

It may produce Facts for:

- detected build system;
- wrapper presence;
- declared project version;
- Java source or target version;
- declared modules;
- dependency coordinates;
- plugin coordinates;
- Spring Boot plugin or parent presence.

Examples:

```text
BUILD_SYSTEM_DETECTED
BUILD_WRAPPER_PRESENT
JAVA_VERSION_DECLARED
PROJECT_VERSION_DECLARED
BUILD_MODULE_DECLARED
DEPENDENCY_DECLARED
BUILD_PLUGIN_DECLARED
```

The Collector parses build descriptors deterministically.

It must not execute arbitrary build scripts or dependency plugins.

For V1, Gradle collection may be limited to statically recognizable declarations.

## 3. SpringCollector

The `SpringCollector` detects objective Spring and Spring Boot characteristics.

It operates only when supported Spring evidence has already been detected from repository files or build descriptors.

It may produce Facts for:

- Spring Boot usage;
- Spring Boot version when explicitly declared;
- Spring Cloud usage;
- Spring Security usage;
- Spring Data usage;
- Spring Web usage;
- Spring Actuator usage;
- application configuration files;
- classes annotated with recognized Spring stereotypes;
- REST controller declarations;
- configuration class declarations.

Examples:

```text
SPRING_BOOT_DETECTED
SPRING_BOOT_VERSION_DECLARED
SPRING_CLOUD_DETECTED
SPRING_SECURITY_DETECTED
SPRING_DATA_DETECTED
SPRING_WEB_DETECTED
SPRING_ACTUATOR_DETECTED
SPRING_CONFIGURATION_FILE_PRESENT
REST_CONTROLLER_DECLARED
SPRING_CONFIGURATION_CLASS_DECLARED
```

V1 uses deterministic textual or syntax-aware detection.

The Collector does not:

- start the Spring application;
- load the Spring context;
- invoke application endpoints;
- infer runtime behavior;
- claim that security is correctly configured.

For example:

```text
Valid Fact:
"Spring Security is declared as a project dependency."

Invalid Fact:
"The application is secure."
```

## 4. DockerCollector

The `DockerCollector` detects containerization-related repository evidence.

It may inspect:

```text
Dockerfile
Dockerfile.*
docker-compose.yml
docker-compose.yaml
compose.yml
compose.yaml
.dockerignore
```

It may produce Facts for:

- Dockerfile presence;
- Docker Compose presence;
- declared Compose services;
- multi-stage build usage;
- non-root user declaration;
- healthcheck declaration;
- exposed ports;
- persistent volume declarations;
- `.dockerignore` presence.

Examples:

```text
DOCKERFILE_PRESENT
DOCKER_COMPOSE_PRESENT
DOCKER_SERVICE_DECLARED
DOCKER_MULTI_STAGE_BUILD_PRESENT
DOCKER_NON_ROOT_USER_DECLARED
DOCKER_HEALTHCHECK_DECLARED
DOCKER_VOLUME_DECLARED
DOCKERIGNORE_PRESENT
```

The Collector does not build or run containers.

It does not infer deployment readiness, security, or production quality.

## 5. DocumentationCollector

The `DocumentationCollector` inventories explicit technical documentation.

It may detect:

- root README files;
- documentation directories;
- Markdown documents;
- ADR directories;
- ADR documents;
- API documentation files;
- architecture documentation;
- contribution guidelines;
- changelogs.

Examples:

```text
README_PRESENT
DOCUMENTATION_DIRECTORY_PRESENT
MARKDOWN_DOCUMENT_PRESENT
ADR_DIRECTORY_PRESENT
ADR_DOCUMENT_PRESENT
API_DOCUMENTATION_PRESENT
ARCHITECTURE_DOCUMENTATION_PRESENT
CONTRIBUTING_GUIDE_PRESENT
CHANGELOG_PRESENT
```

The Collector may extract deterministic metadata such as:

- relative path;
- title or first heading;
- file size;
- last Git modification commit when already available.

It must not evaluate whether documentation is correct, sufficient, outdated, or high quality.

Those conclusions belong to future Observation rules.

## 6. TestStructureCollector

The `TestStructureCollector` detects the existence and structure of automated tests.

It may produce Facts for:

- test source directory presence;
- detected test files;
- detected test framework dependencies;
- integration-test naming conventions;
- Testcontainers usage;
- test resource directories;
- test counts grouped by supported category.

Examples:

```text
TEST_SOURCE_DIRECTORY_PRESENT
TEST_FILE_PRESENT
TEST_FRAMEWORK_DECLARED
INTEGRATION_TEST_FILE_PRESENT
TESTCONTAINERS_DECLARED
TEST_RESOURCE_DIRECTORY_PRESENT
```

The Collector does not execute tests.

It does not calculate actual test coverage.

It must not claim that the project is tested correctly or sufficiently.

For example:

```text
Valid Fact:
"The repository contains 24 files matching the configured unit-test convention."

Invalid Fact:
"The project has good test coverage."
```

## Fact Granularity

Collectors must avoid both extremes:

### Excessively coarse Facts

```text
"The project uses Spring."
```

This may lose useful traceability.

### Excessively granular Facts

```text
"Line 17 of file X contains annotation Y."
```

This would create excessive persistence volume and noisy AI context.

V1 should favor meaningful technical units such as:

- one dependency declaration;
- one module declaration;
- one REST controller declaration;
- one documentation document;
- one Docker service;
- one summarized test category.

Facts should remain independently understandable and traceable.

## Duplicate Prevention

A Collector must produce a stable deterministic fingerprint for every Fact candidate.

Conceptually, the fingerprint may derive from:

```text
collectorVersion
+ factType
+ normalized content
+ normalized evidence references
+ resolved revision
```

The persistence layer uses this fingerprint or an equivalent deterministic identity strategy to prevent duplicate Facts inside the same Analysis.

Re-running a Collector against the same revision must not create duplicate Facts.

Facts from separate Analyses remain historically distinct according to ADR-016.

## Collector Ordering

Collectors may require deterministic evidence produced by repository inspection, but they must not depend directly on the output of another Collector through service-to-service calls.

The collection workflow defines a stable execution order:

```text
GitCollector
    ‚Üì
RepositoryMetadataCollector
    ‚Üì
BuildCollector
    ‚Üì
SpringCollector
    ‚Üì
DockerCollector
    ‚Üì
DocumentationCollector
    ‚Üì
TestStructureCollector
```

This order is used for reproducibility and diagnostics.

A Collector may inspect the same workspace files as another Collector when required by its own responsibility.

Shared low-level parsing utilities may be introduced to avoid duplicated technical parsing, but they must not merge Collector responsibilities.

## Failure Policy

Collector failures are classified as either fatal or non-fatal.

### Fatal failure

Examples:

- workspace inaccessible;
- resolved revision unavailable;
- repository synchronization inconsistency;
- collection context invalid.

A fatal failure stops collection and transitions the Analysis to `FAILED`.

### Non-fatal collector failure

Examples:

- one unsupported Gradle expression;
- one malformed optional documentation file;
- one unrecognized Docker Compose extension.

A non-fatal failure:

- produces a structured warning;
- does not invent a Fact;
- allows the remaining Collectors to continue.

The workflow must expose sufficient diagnostics to identify which Collector produced each warning.

## Security Boundaries

Collectors inspect untrusted repositories.

Therefore, Collectors must not:

- execute repository scripts;
- execute Maven or Gradle;
- run shell commands defined by the repository;
- execute tests;
- launch containers;
- follow symbolic links outside the workspace;
- read files outside the synchronized workspace;
- persist secrets detected in files;
- include full sensitive configuration values in Facts.

Potential secrets must be redacted or excluded.

Repository paths must be normalized and checked against the workspace root before reading.

## Performance Boundaries

Collectors must remain suitable for regular repeated analyses.

V1 introduces configurable limits for:

- maximum number of inspected files;
- maximum individual file size;
- maximum total bytes read per Collector;
- maximum Facts produced per type;
- excluded directories;
- execution timeout per Collector.

Default excluded directories should include generated or dependency content such as:

```text
.git
target
build
node_modules
dist
out
coverage
.venv
venv
```

Limits must produce explicit warnings rather than silently truncating the result.


All initial Collectors are enabled by default for Git repository Sources.

Future user-level Collector profiles are outside the scope of this ADR.

## Observation Boundary

ADR-024 intentionally introduces no speculative Observation rules.

Collectors stop at Facts.

Examples of possible future Observations that are explicitly outside this increment include:

- the project follows a microservice architecture;
- documentation does not cover recent modules;
- the repository uses consistent containerization practices;
- test coverage appears insufficient;
- security configuration is incomplete.

Such rules must be implemented separately in the `ObservationEngine`, supported by explicit Facts and covered by deterministic tests.

## AnalysisContext Integration

Facts produced by the new Collectors become eligible for the existing deterministic `AnalysisContext`.

The context builder remains responsible for:

- selection;
- ordering;
- limits;
- relevance policies;
- serialization into immutable snapshots.

Collectors must not decide which Facts are sent to the AI Engine.

## Initial Implementation Scope

This increment includes:

- the common Collector contract;
- immutable collection context and result models;
- Collector type and version metadata;
- `RepositoryMetadataCollector`;
- `BuildCollector` with Maven and bounded Gradle support;
- `SpringCollector`;
- `DockerCollector`;
- `DocumentationCollector`;
- `TestStructureCollector`;
- deterministic Fact fingerprints;
- failure and warning handling;
- configurable file and Fact limits;
- secure workspace path validation;
- unit tests for every Collector;
- orchestration tests;
- integration with the existing collection workflow;
- updated documentation.

This increment excludes:

- Java AST or compiler-based semantic analysis;
- complete source-code dependency graphs;
- runtime application inspection;
- test execution;
- coverage measurement;
- vulnerability scanning;
- secret scanning;
- GitHub or GitLab APIs;
- pull requests and issues;
- CI/CD provider APIs;
- Terraform or Kubernetes Collectors;
- automatic Observation rules;
- direct AI interpretation of repository files.

## Alternatives Considered

### One generic repository Collector

Rejected because:

- it would combine unrelated responsibilities;
- it would become difficult to test and evolve;
- adding new technology support would continuously expand one component;
- Fact ownership and diagnostics would become unclear.

### Execute project tooling for richer results

Run Maven, Gradle, tests, Docker or application startup during collection.

Rejected for V1 because:

- repositories are untrusted;
- commands may execute arbitrary code;
- execution would make collection slower and less deterministic;
- environment-specific failures would reduce reproducibility.

### Send repository files directly to the LLM

Rejected because:

- it violates the deterministic-first architecture;
- it would weaken traceability;
- token consumption would become uncontrolled;
- project knowledge retrieval would move outside the Core.

### Introduce every planned Collector immediately

Rejected because:

- the V1 needs a useful but bounded context;
- unsupported technologies would add speculative complexity;
- collector value should be validated using real repository analyses.

## Consequences

### Positive

- Produces a substantially richer deterministic project description.
- Enables meaningful first analyses of real repositories.
- Preserves strict separation between Facts, Observations and AI interpretation.
- Keeps each technical concern independently testable.
- Improves evidence traceability.
- Provides predictable limits for large repositories.
- Avoids executing untrusted project code.
- Establishes a versioned foundation for future collector evolution.

### Negative

- Adds several specialized components.
- Static parsing cannot capture every dynamic build configuration.
- Fact volume will increase and require careful context selection.
- Collector versioning and diagnostics require additional metadata.
- New technologies will require new Collectors or parser extensions.
- Determining appropriate Fact granularity will require iteration with real repositories.

## Architectural Principle

DevLog AI follows this rule:

```text
Collectors collect objective evidence.
Observation Engine derives deterministic patterns.
AI Engine interprets the resulting context.
Humans validate proposed knowledge.
```

Or, in its simplified form:

```text
Collect
    ‚Üì
Observe
    ‚Üì
Interpret
    ‚Üì
Validate
    ‚Üì
Knowledge
```

The fundamental constraint remains:

> **A Collector describes what can be objectively detected in a repository. It never decides what that evidence means.**
