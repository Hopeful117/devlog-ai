# ADR-010 - Asynchronous AI Processing with REST Communication

## Status

Accepted

## Date

2026-07-21

## Context

DevLog AI separates deterministic project analysis from AI interpretation.

The Java Core prepares structured context and delegates AI processing to the Python AI Engine.

AI processing may take significantly longer than standard synchronous application operations due to:

* context preparation,
* prompt construction,
* model inference,
* external AI provider calls,
* multiple AI processing steps.

The system therefore should not require the main application workflow to remain blocked while AI processing is performed.

At the same time, introducing a distributed event-driven architecture with message brokers would add significant infrastructure and operational complexity to the V1.

DevLog AI needs a communication mechanism that:

* keeps the Core and AI Engine clearly separated,
* supports long-running AI processing,
* remains simple to develop and operate,
* can evolve toward more advanced distributed processing if future requirements justify it.

## Decision

DevLog AI will use asynchronous processing for AI tasks while using REST as the initial communication protocol between the Java Core and Python AI Engine.

The Core remains responsible for initiating and tracking the business workflow.

The AI Engine processes the requested AI task asynchronously and returns the result to the Core.

The conceptual workflow is:

```text id="q0w4xe"
User / Trigger
      │
      ▼
Java Core
      │
      ├── Create Analysis
      ├── Generate Facts
      ├── Generate Observations
      └── Create AI Task
             │
             │ REST
             ▼
      Python AI Engine
             │
             ├── Process AI Task
             ├── Model Inference
             └── Generate Result
             │
             │ REST
             ▼
         Java Core
             │
             ▼
    ValidatableProposal
             │
             ▼
      Human Validation
```

The AI task is therefore asynchronous from the perspective of the business workflow, even though the inter-service communication initially uses HTTP REST.

The Core must not remain blocked while waiting for the complete AI processing pipeline.

## Responsibilities

### Java Core

The Core is responsible for:

* creating AI tasks,
* providing structured context,
* initiating AI processing,
* tracking the business state of the task,
* receiving AI results,
* persisting proposals,
* associating proposals with the relevant analysis,
* initiating the validation workflow.

### Python AI Engine

The AI Engine is responsible for:

* receiving AI task requests,
* executing AI processing,
* managing its internal processing state,
* performing model inference,
* generating structured AI results,
* returning the result to the Core.

The AI Engine does not own the overall business workflow.

## Communication Model

The initial V1 communication model is REST-based:

```text id="v1n4j7"
Core
  │
  │ POST /ai/tasks
  ▼
AI Engine
  │
  │ processing
  │
  │ POST /ai/tasks/{id}/result
  ▼
Core
```

The exact API design may evolve, but the business workflow must remain asynchronous.

The Core should therefore be able to represent states such as:

```text id="x6q8v2"
CREATED
   │
   ▼
SUBMITTED
   │
   ▼
PROCESSING
   │
   ▼
COMPLETED
```

Additional failure and retry states may be introduced as the implementation evolves.

## Alternatives Considered

### Fully synchronous REST communication

The Core would call the AI Engine and wait for the complete AI processing result before continuing.

Rejected because:

* AI inference may be slow,
* long-running HTTP requests are fragile,
* the Core would remain blocked,
* retries and failures become harder to manage,
* user-facing workflows would be unnecessarily coupled to AI processing time.

### Event-driven communication with a message broker

Use Kafka, RabbitMQ, or another messaging platform for communication between the Core and AI Engine.

Rejected for V1 because:

* it introduces additional infrastructure,
* operational complexity increases,
* message delivery and retry semantics require additional design,
* the current system does not yet justify the complexity of distributed event processing.

This approach may be reconsidered if future scale or reliability requirements justify it.

### Direct synchronous communication with polling

The Core would create a task and repeatedly poll the AI Engine for its status.

Rejected as the default approach because:

* it creates unnecessary network traffic,
* it complicates task completion handling,
* it increases coupling between the Core and AI Engine.

A polling mechanism may still be used internally if required by a future AI provider integration, but it is not the primary inter-service workflow.

## Consequences

### Positive

* AI processing does not block the main business workflow.
* REST keeps the initial architecture simple.
* The Core and AI Engine remain clearly separated.
* The system can support long-running AI tasks.
* The architecture can evolve toward messaging later.
* V1 avoids premature distributed infrastructure.

### Negative

* Asynchronous processing requires explicit task state management.
* Failure and retry handling must be designed.
* REST does not inherently provide message delivery guarantees.
* The system may eventually need a messaging infrastructure if processing volume increases significantly.
