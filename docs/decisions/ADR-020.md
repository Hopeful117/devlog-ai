# ADR-020 - AI Task Result Callback and Proposal Creation

## Status

Accepted

## Date

2026-07-21

## Context

ADR-010 established asynchronous REST communication between the Java Core and the Python AI Engine.

ADR-017 introduced the persistent `AiTask` lifecycle.

ADR-018 introduced the `AnalysisWorkflowService`.

ADR-019 defined the submission contract from the Core to the AI Engine.

The remaining missing piece is the return path from the AI Engine back to the Core once AI processing has completed.

The callback must preserve the architectural principles already established:

* the Java Core remains the source of truth;
* the AI Engine performs interpretation only;
* trusted project knowledge is never created directly by the AI Engine;
* all AI outputs remain proposals until validated.

## Decision

The AI Engine will notify the Java Core using an asynchronous REST callback.

The callback is initiated by the AI Engine after task execution completes.

```text
Java Core
    │
POST /api/v1/ai/tasks
    ▼
AI Engine
    │
202 Accepted
    │
AI processing
    │
POST /api/v1/ai/tasks/{correlationId}/result
    ▼
Java Core
```

The callback delivers structured AI results.

The Core remains responsible for interpreting these results as business objects.

## Callback Contract

The AI Engine sends:

```text
AiTaskResultRequest
├── correlationId
├── externalJobId
├── status
├── completedAt
├── proposals[]
└── error
```

The callback must contain the original `correlationId` generated by the Core.

The callback never references internal database identifiers.

## Proposal Contract

Each proposal contains:

```text
AiProposalResult
├── type
├── payload
├── confidence
├── supportingFactIds
├── supportingObservationIds
└── evidenceReferences
```

The payload represents an AI interpretation.

It is not trusted project knowledge.

## Core Responsibilities

When receiving the callback, the Java Core must:

1. locate the corresponding `AiTask` using the `correlationId`;
2. validate the current task state;
3. validate the callback payload;
4. persist one `ValidatableProposal` per AI proposal;
5. transition the `AiTask` to `COMPLETED`;
6. return an acknowledgement.

The callback must not create:

* `Decision`;
* `KnowledgeEvent`;
* `Insight`;
* `Documentation`;
* any other trusted domain object.

Only `ValidatableProposal` objects are created.

## AI Engine Responsibilities

The AI Engine is responsible for:

* executing the AI task;
* generating structured proposals;
* reporting technical execution status;
* invoking the callback.

The AI Engine never modifies project knowledge directly.

## Idempotency

The callback endpoint must be idempotent.

Receiving the same callback multiple times must not:

* duplicate proposals;
* duplicate state transitions;
* corrupt task history.

If the task has already reached a terminal state, duplicate callbacks should be safely ignored or acknowledged according to implementation policy.

## Transaction Boundary

The following operations must occur atomically:

* proposal creation;
* task state transition.

Either both succeed or neither is persisted.

## Failure Handling

If:

* the task cannot be found;
* the callback payload is invalid;
* proposal persistence fails;

the callback must return an appropriate error response and preserve the consistency of the Core.

Technical failures should be recorded for observability.

## Initial Implementation Scope

The first implementation increment includes:

* callback REST endpoint;
* request/response DTOs;
* callback service;
* creation of simulated `ValidatableProposal` objects;
* `AiTask` transition to `COMPLETED`;
* unit tests;
* integration tests.

The first increment explicitly excludes:

* real LLM integration;
* proposal scoring improvements;
* automatic retries;
* knowledge promotion;
* human validation workflow changes.

## Alternatives Considered

### AI Engine writes directly into the database

Rejected because it violates the Core ownership model and breaks ADR-012.

### AI Engine creates trusted knowledge

Rejected because AI outputs must always remain proposals until validated.

### Polling instead of callbacks

Rejected because asynchronous callbacks reduce latency and avoid unnecessary polling traffic.

## Consequences

### Positive

* Completes the distributed workflow.
* Preserves the Core as the source of truth.
* Keeps AI outputs isolated from trusted knowledge.
* Supports future AI providers without changing the Core.
* Enables end-to-end integration testing.

### Negative

* Introduces callback management.
* Requires idempotent processing.
* Adds another network interaction between services.
