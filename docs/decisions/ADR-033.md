# ADR-033 — Knowledge Selection Engine

## Status

Accepted

## Date

2026-07-22

---

# Context

Previous ADRs established the complete deterministic knowledge pipeline and the prompt generation architecture.

```text
Repository
    ↓
Collectors
    ↓
Facts
    ↓
Observations
    ↓
Project Profile
    ↓
AnalysisContext
    ↓
Intent
    ↓
User Guidance
    ↓
Prompt Builder
    ↓
LLM Provider
    ↓
InsightProposal
```

As the project evolves, the amount of available knowledge continuously increases.

Future analyses may contain:

- hundreds of Facts;
- hundreds of Observations;
- Project Profiles;
- Analysis Diagnostics;
- Validated Insights;
- historical Analyses;
- Architecture Decision Records;
- future knowledge sources.

Sending the complete project knowledge to an LLM is neither efficient nor desirable.

DevLog AI therefore requires a deterministic mechanism responsible for selecting the most relevant knowledge before prompt construction.

---

# Decision

DevLog AI introduces a dedicated **Knowledge Selection Engine**.

The Knowledge Selection Engine is responsible for selecting the relevant project knowledge required by a specific Intent.

It produces an immutable **SelectedKnowledge** object consumed by the Prompt Builder.

```text
Knowledge Base
        │
        ▼
Knowledge Selection Engine
        │
        ▼
Selected Knowledge
        │
        ▼
Prompt Builder
```

---

# Responsibilities

The Knowledge Selection Engine is responsible for:

- selecting relevant knowledge;
- applying deterministic selection rules;
- respecting the business objective defined by the Intent;
- preserving deterministic traceability;
- enforcing knowledge budget constraints;
- removing redundant information;
- preparing provider-independent knowledge for prompt construction.

---

# Non Responsibilities

The Knowledge Selection Engine is not responsible for:

- generating prompts;
- prompt engineering;
- provider communication;
- AI inference;
- summarizing knowledge;
- rewriting knowledge;
- generating Insight Proposals.

Those responsibilities belong to the Prompt Builder or AI Engine.

---

# Knowledge Sources

The engine may select knowledge from:

```text
Facts

Observations

Project Profiles

Analysis Diagnostics

Validated Insights

Historical Analyses

Architecture Decision Records

Future Knowledge Sources
```

The supported sources may grow over time without modifying the Prompt Builder.

---

# SelectedKnowledge

The engine produces an immutable provider-independent object.

Conceptually:

```text
SelectedKnowledge
├── intent
├── projectProfile
├── selectedObservations
├── selectedFacts
├── diagnostics
├── selectedInsights
├── selectionMetadata
└── selectionDigest
```

SelectedKnowledge becomes the only project knowledge visible to the Prompt Builder.

---

# Architectural Boundary

The Prompt Builder must never access repositories, databases or project knowledge directly.

Its only knowledge input is:

```text
SelectedKnowledge
```

This separation guarantees that:

- prompt engineering remains independent from knowledge selection;
- selection rules evolve without modifying prompt templates;
- future providers receive identical knowledge.

---

# Deterministic Selection

Knowledge selection must always be deterministic.

Given:

- the same Analysis;
- the same Intent;
- the same User Guidance;
- the same Selection Engine version;

the engine must always produce the same SelectedKnowledge.

Selection must never depend on:

- randomness;
- provider behaviour;
- execution timing.

---

# Selection Pipeline

Conceptually:

```text
Knowledge Base
        │
        ▼
Intent
        │
        ▼
Candidate Selection
        │
        ▼
Filtering
        │
        ▼
Ranking
        │
        ▼
Knowledge Budget
        │
        ▼
SelectedKnowledge
```

Each stage is deterministic.

---

# Intent-driven Selection

Intent determines the knowledge required.

Example:

```text
describe-project-v1
```

may prioritize:

- Project Profile;
- architectural Observations;
- selected Facts;
- diagnostics.

Another Intent may use a completely different selection strategy.

The Prompt Builder never performs this selection.

---

# Knowledge Budget

The Knowledge Selection Engine enforces a provider-independent Knowledge Budget.

The budget expresses the maximum amount of knowledge that may be forwarded to prompt construction.

The Knowledge Budget is a business concept.

It is intentionally independent from:

- tokens;
- context windows;
- provider limitations.

Provider-specific adaptations remain Provider responsibilities.

---

# Redundancy Elimination

When multiple knowledge objects express the same information, deterministic rules may retain only the most relevant representation.

For example:

```text
Fact

↓

Observation

↓

Project Profile
```

The engine may choose the highest semantic level when appropriate.

Knowledge removal must never be arbitrary.

---

# Future Evolution

The initial implementation relies exclusively on deterministic rules.

Future versions may introduce additional ranking signals such as:

- validated Insights;
- historical usage;
- user preferences;
- project evolution.

Those future improvements must remain deterministic unless explicitly superseded by a future ADR.

---

# Selection Metadata

SelectedKnowledge contains metadata describing the selection process.

Conceptually:

```text
SelectionMetadata
├── selectionVersion
├── appliedRules
├── selectedKnowledgeCount
├── discardedKnowledgeCount
├── knowledgeBudget
└── completeness
```

This metadata supports diagnostics and replay.

---

# Selection Digest

The engine computes a deterministic digest over SelectedKnowledge.

Example:

```text
SHA-256(
    orderedSelectedKnowledge
)
```

The digest supports:

- reproducibility;
- diagnostics;
- replay validation;
- prompt traceability.

---

# Failure Policy

Selection fails when:

- mandatory knowledge is unavailable;
- Intent requirements cannot be satisfied;
- selection rules become inconsistent;
- deterministic ordering cannot be guaranteed.

In this case:

```text
AI Task → FAILED

Analysis → FAILED
```

No Prompt is constructed.

---

# Testability

The Knowledge Selection Engine must be fully testable without:

- LLM providers;
- Prompt Builder;
- network access.

Tests must verify:

- deterministic ordering;
- identical outputs for identical inputs;
- Intent-specific selection;
- knowledge budget enforcement;
- redundancy elimination;
- selection metadata;
- digest stability.

---

# Consequences

## Positive

- Clear separation between knowledge selection and prompt engineering.
- Deterministic AI context preparation.
- Better scalability.
- Easier provider replacement.
- Stable replay capabilities.
- Future-ready architecture.

## Negative

- Additional architectural component.
- Additional versioning.
- Requires deterministic ranking rules.

---

# Alternatives Considered

## Prompt Builder selecting knowledge

Rejected.

Prompt construction and knowledge selection are different responsibilities.

---

## Sending the complete AnalysisContext

Rejected.

Does not scale and weakens prompt quality.

---

## Provider-driven retrieval

Rejected.

Knowledge selection belongs to the business domain, not to external AI providers.

---

# Architectural Principle

Collectors determine what exists.

Observation Engine determines what can be objectively concluded.

Project Profile determines the project's identity.

Knowledge Selection Engine determines which knowledge is relevant for a specific Intent.

Prompt Builder determines how that knowledge is presented to an LLM.

LLM Providers determine how the Prompt is executed.

Each component owns one responsibility.
