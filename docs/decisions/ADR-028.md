# ADR-028 — Intent-driven Insight Generation

## Status

Accepted

## Date

2026-07-22

---

# Context

Previous ADRs established the deterministic knowledge pipeline.

```text
Repository
    ↓
Workspace
    ↓
Collectors
    ↓
Facts
    ↓
Observation Engine
    ↓
Project Profile
    ↓
AnalysisContext
```

At this stage DevLog AI possesses a deterministic understanding of a software project.

However, the AI Engine still lacks one essential element:

> Why is the user asking for an AI analysis?

Calling an LLM without a clearly defined objective would produce inconsistent outputs, unstable prompts and poorly scoped responses.

The AI Engine must therefore be driven by explicit user intent rather than generic repository analysis.

---

# Decision

DevLog AI introduces **Intent** as a first-class business concept.

Intent represents the user's objective.

It determines:

- why the AI is called;
- what information should be emphasized;
- what kinds of Insights may be generated;
- the expected response structure;
- the prompt template used to communicate with the provider.

The AI Engine never receives a free-form request.

Instead, it receives:

```text
Intent

+

AnalysisContext

↓

Prompt Builder

↓

Prompt

↓

LLM

↓

InsightProposal[]
```

---

# Intent Definition

An Intent represents a deterministic contract between the user request and the AI Engine.

It answers:

> What does the user expect?

It does not describe:

- how the provider should implement the reasoning;
- how the prompt is rendered;
- how the Insight will eventually be displayed.

---

# Intent Model

Conceptually:

```text
Intent
├── id
├── version
├── objective
├── supportedInsightTypes
├── constraints
├── outputSchema
└── promptTemplate
```

Every Intent is versioned.

---

# Intent Versioning

Intent versions are immutable.

Examples:

```text
describe-project-v1

generate-readme-v1

architecture-overview-v1

module-documentation-v1
```

Changing:

- prompt semantics;
- expected Insight types;
- output schema;

requires a new Intent version.

---

# Prompt Builder

The Prompt Builder is responsible for transforming:

```text
Intent

+

AnalysisContext
```

into

```text
Provider Prompt
```

The Prompt itself is therefore an implementation detail.

Intent remains the business contract.

---

# Separation of Responsibilities

Intent defines:

- the objective;
- expected Insight categories;
- constraints.

Prompt Builder defines:

- provider-specific formatting;
- prompt wording;
- serialization.

Provider defines:

- AI execution.

This separation allows Prompt Builder implementations to evolve independently from business semantics.

---

# Insight Generation

Intent does not generate documentation.

Intent generates Insight Proposals.

Documentation becomes one possible application of validated Insights.

The business workflow therefore remains:

```text
Intent

↓

InsightProposal

↓

Human Validation

↓

Trusted Knowledge

↓

User Output
```

---

# Human in the Loop

Intent never bypasses human validation.

Every generated Insight remains:

- proposed;
- reviewable;
- rejectable;
- traceable.

Intent cannot directly create trusted project knowledge.

---

# Deterministic Context

Intent never accesses repository files directly.

Its only knowledge source is:

```text
AnalysisContext
```

which already contains:

- Facts;
- Observations;
- Project Profile;
- Validated Knowledge;
- Analysis metadata.

The provider must never reconstruct project understanding independently.

---

# Supported Insight Types

Every Intent explicitly declares the Insight types it may generate.

Example:

```text
Describe Project

↓

PROJECT_PRESENTATION

ARCHITECTURE_DESCRIPTION

TECHNOLOGY_DESCRIPTION
```

Another Intent:

```text
Generate README

↓

INSTALLATION

USAGE

REQUIREMENTS

PROJECT_PRESENTATION
```

The provider should never generate unsupported Insight categories.

---

# Output Schema

Every Intent defines the expected structured output.

The AI provider always returns structured Insight proposals.

Free-form responses are not part of the business contract.

---

# Initial Intent Set

The initial implementation introduces a limited deterministic set.

```text
describe-project-v1

generate-readme-v1

architecture-overview-v1
```

Additional Intents may be introduced without modifying the deterministic pipeline.

---

# Intent Lifecycle

```text
User Request

↓

Intent Resolution

↓

AnalysisContext

↓

Prompt Builder

↓

AI Provider

↓

InsightProposal[]

↓

Human Validation

↓

Knowledge
```

---

# Architectural Principle

Intent defines **why** the AI is called.

AnalysisContext defines **what** the AI knows.

Prompt Builder defines **how** that knowledge is presented to the provider.

The provider defines **how** the reasoning is performed.

The AI Engine never decides its own objective.

The objective always comes from the user through an explicit Intent.

---

# Consequences

## Positive

- Explicit business objective.
- Stable prompt architecture.
- Easier provider replacement.
- Better prompt versioning.
- Better testability.
- Strong separation between business and implementation.
- Scalable Insight generation.
- Human validation preserved.

## Negative

- Additional business abstraction.
- Intent catalog to maintain.
- Prompt Builder becomes an additional component.

---

# Alternatives Considered

## Free-form user prompts

Rejected.

Business semantics become coupled to prompt wording.

---

## Provider-specific prompt definitions

Rejected.

Prompts should remain implementation details.

---

## One generic Intent

Rejected.

Different user objectives require different Insight contracts.
