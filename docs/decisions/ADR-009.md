# ADR-009 - Separation of Deterministic Analysis and AI Interpretation

## Status

Accepted

## Date

2026-07-21

## Context

DevLog AI combines deterministic software engineering analysis with artificial intelligence capabilities.

The system needs to distinguish between information that can be reliably established from project data and conclusions that require contextual interpretation.

Some information can be determined objectively:

* a dependency was added,
* a file was modified,
* a class was created,
* an API endpoint was removed,
* a technology migration occurred,
* code complexity increased,
* documentation is missing or outdated.

Other conclusions require interpretation:

* why a technical decision was made,
* what the developer intended,
* whether an architectural change was strategic,
* what problem a change was intended to solve,
* whether multiple technical changes represent a broader engineering decision.

Allowing the AI Engine to perform deterministic analysis would introduce unnecessary uncertainty and duplicate responsibilities that can be handled reliably by the Core.

Conversely, attempting to implement all contextual interpretation deterministically would limit DevLog AI's ability to understand engineering intent.

This decision refines the hybrid architecture established by ADR-003.

## Decision

DevLog AI will strictly separate deterministic analysis from AI interpretation.

The Java Core is responsible for all analysis that can be performed reliably using deterministic rules and structured project data.

The Python AI Engine is responsible for interpretation and reasoning where deterministic rules are insufficient.

The boundary is:

```text id="l7j6di"
                    JAVA CORE
                         │
                         ▼
                   Evidence
                         │
                         ▼
                      Facts
                         │
                         ▼
              Deterministic Analysis
                         │
                         ▼
                   Observations
                         │
                         ▼
                Structured Context
                         │
                         │
                         ▼
                  PYTHON AI ENGINE
                         │
                         ▼
                  Interpretation
                         │
                         ▼
                  AI Proposals
```

### Java Core Responsibilities

The Core is responsible for:

* repository analysis,
* objective fact extraction,
* deterministic observations,
* domain rules,
* structured project knowledge,
* evidence traceability,
* context preparation for AI tasks.

Examples include:

```text id="j6v3d2"
FILE_ADDED
DEPENDENCY_ADDED
CLASS_REMOVED
API_ENDPOINT_ADDED
ARCHITECTURE_CHANGED
REFACTORING_DETECTED
DOCUMENTATION_GAP_DETECTED
```

These results must be derived from explicit rules and available project data.

### AI Engine Responsibilities

The AI Engine is responsible for:

* semantic interpretation,
* contextual reasoning,
* intent inference,
* hypothesis generation,
* identifying relationships that cannot be reliably established deterministically,
* proposing engineering decisions,
* proposing engineering events,
* generating contextual content.

Examples include:

```text id="m4t9g1"
"The architecture appears to have moved toward asynchronous processing."

"This change was likely motivated by scalability concerns."

"The team may have introduced this abstraction to reduce coupling."
```

Such conclusions are considered AI interpretations and are not authoritative by default.

## Core Architectural Rule

DevLog AI follows the following principle:

> If the Core can determine an answer reliably and deterministically, the Core must own that responsibility.

The AI Engine should only be used when interpretation, contextual reasoning, or semantic understanding provides additional value.

Conceptually:

```text id="9r1x4v"
Can the information be reliably determined?
                 │
        ┌────────┴────────┐
        │                 │
       YES                NO
        │                 │
        ▼                 ▼
   Java Core          AI Engine
```

The boundary is based on responsibility rather than implementation convenience.

A task must not be moved to Python merely because it is easier to implement there.

## AI Output

AI-generated interpretations are not considered authoritative project knowledge.

They are returned to the Core as proposals.

```text id="e7b3q1"
AI Interpretation
       │
       ▼
ValidatableProposal
       │
       ▼
Human Validation
       │
       ├── Accepted
       └── Rejected
```

The Core remains responsible for deciding how the proposal affects trusted project knowledge.

## Alternatives Considered

### AI performs all project analysis

Allow Python to directly analyze repositories and determine both objective facts and interpretations.

Rejected because:

* deterministic results would become dependent on AI behavior,
* outputs could be inconsistent,
* traceability would be reduced,
* the AI Engine would assume responsibilities belonging to the Core,
* AI inference would be unnecessarily used for objectively determinable information.

### Core performs all analysis

Implement all analysis and interpretation using deterministic rules.

Rejected because:

* engineering intent cannot always be inferred from technical changes,
* contextual reasoning is difficult to encode entirely through predefined rules,
* the system would lose the benefits of AI-assisted interpretation.

### Core and AI Engine perform overlapping analysis

Allow both services to independently analyze the same information.

Rejected because:

* responsibilities would become ambiguous,
* results could conflict,
* duplicated processing would increase complexity,
* it would be difficult to determine which result is authoritative.

## Consequences

### Positive

* Clear responsibility boundaries.
* Objective information remains deterministic and traceable.
* AI is used where it provides the most value.
* AI models can evolve independently from the deterministic Core.
* The system reduces unnecessary AI hallucination risks.
* The architecture remains consistent with ADR-003.
* Human validation can focus on AI interpretation rather than revalidating deterministic Core output.

### Negative

* The Core requires a sufficiently capable deterministic analysis layer.
* Communication between the Core and AI Engine is necessary.
* The boundary between deterministic analysis and interpretation may occasionally require architectural judgment.
* Some complex observations may require progressively more sophisticated deterministic rules.
