# ADR-008 - Generic Facts with Strongly Typed Payloads

## Status

Accepted

## Date

2026-07-21

## Context

DevLog AI needs to represent objective information discovered during deterministic project analysis.

The system may identify many different categories of facts, including:

* commits,
* file changes,
* dependency changes,
* code changes,
* architecture changes,
* test changes,
* documentation changes,
* technology changes.

Creating a dedicated persistent entity for every possible fact type would lead to a growing number of JPA entities and make the domain model increasingly difficult to maintain.

At the same time, storing facts as completely unstructured key-value pairs or arbitrary JSON would reduce type safety and make the deterministic analysis layer less explicit and reliable.

The system therefore needs a model that combines:

* a common persistence and traceability model,
* explicit fact types,
* strongly typed domain data.

## Decision

DevLog AI will use a generic `Fact` entity as the common persistence and traceability boundary for deterministic facts.

Each `Fact` will contain a type identifying the category of fact and a strongly typed payload representing the associated data.

Conceptually:

```text
Fact
├── id
├── analysis
├── type
├── detectedAt
├── source
├── evidenceReferences
└── payload
```

The payload will be represented by an explicit Java type corresponding to the fact category.

For example:

```text
Fact
│
├── type = COMMIT
│   └── CommitFactData
│
├── type = FILE_CHANGE
│   └── FileChangeFactData
│
├── type = DEPENDENCY_CHANGE
│   └── DependencyChangeFactData
│
├── type = CODE_METRIC
│   └── CodeMetricFactData
│
└── type = DOCUMENTATION_CHANGE
    └── DocumentationChangeFactData
```

The system will therefore avoid creating a dedicated JPA entity for every fact category.

The model should remain generic at the persistence level while remaining explicit and strongly typed at the application and domain level.

The same architectural principle may be applied to deterministic `Observation` entities when appropriate.

## Fact Design Principle

Facts represent objective and deterministic information.

They should describe what the system can reliably establish rather than what the system believes the information means.

For example:

```text
DEPENDENCY_ADDED
technology = Kafka
```

is a fact.

By contrast:

```text
Kafka was introduced to improve scalability.
```

is an interpretation and should not be represented as a deterministic fact.

The distinction between objective facts and interpretation is maintained throughout the analysis pipeline.

```text
Evidence
    │
    ▼
Fact
    │
    ▼
Observation
    │
    ▼
AI Interpretation
```

## Alternatives Considered

### Dedicated entity for each Fact type

Create separate entities such as:

```text
CommitFact
FileChangeFact
DependencyChangeFact
ClassChangeFact
CodeMetricFact
DocumentationChangeFact
```

Rejected because:

* the number of entities would grow with every new fact category,
* persistence mappings would become more complex,
* common traceability behavior would be duplicated,
* introducing a new fact type would require additional infrastructure.

### Fully unstructured key-value Facts

Represent every fact using a generic structure such as:

```text
Fact
├── type
├── key
└── value
```

Rejected because:

* type safety would be reduced,
* domain semantics would become implicit,
* validation would be more difficult,
* consumers would need to interpret arbitrary values,
* deterministic processing would become less reliable.

### Arbitrary JSON payloads without typed domain models

Store all fact payloads as unvalidated JSON structures.

Rejected because:

* the domain model becomes implicit,
* compile-time guarantees are lost,
* payload evolution becomes harder to control,
* validation becomes dependent on runtime behavior.

JSON or another generic serialization format may still be used as a persistence or transport mechanism where appropriate, but the application model should remain strongly typed.

## Consequences

### Positive

* Provides a consistent persistence model for facts.
* Preserves strong typing in the application layer.
* Avoids an excessive number of persistent entities.
* Makes fact categories explicit.
* Allows new fact types to be introduced without redesigning the entire persistence model.
* Supports reliable deterministic analysis.
* Maintains a clear separation between facts and AI interpretations.

### Negative

* Generic payload persistence requires explicit serialization and deserialization logic.
* Type validation must be implemented carefully.
* The persistence representation may be less straightforward than dedicated relational tables.
* Additional mapping logic may be required between persisted payloads and strongly typed Java models.
