# ADR-015 - Deterministic Retrieval for V1

## Status

Accepted

## Date

2026-07-21

## Context

DevLog AI will eventually benefit from a hybrid retrieval strategy combining deterministic and semantic retrieval, as established by ADR-014.

However, the V1 system does not yet have enough project knowledge volume or real-world usage data to justify introducing vector search infrastructure.

Introducing embeddings and a Vector Store at this stage would add significant complexity, including:

* embedding model selection,
* embedding generation,
* embedding lifecycle management,
* vector storage,
* indexing,
* synchronization,
* semantic search,
* relevance ranking,
* potential re-embedding when models change.

The system also does not yet have sufficient real-world data to determine:

* which entities should be embedded,
* the appropriate embedding granularity,
* which retrieval queries require semantic search,
* how much historical knowledge must be retrieved,
* which ranking strategy provides the best results.

DevLog AI therefore needs to prioritize a simple and reliable retrieval mechanism for V1 while preserving the ability to introduce semantic retrieval later.

## Decision

DevLog AI V1 will use deterministic retrieval only.

The Java Core will retrieve relevant project knowledge using structured relationships, metadata, and explicit filtering.

No Vector Store or semantic retrieval infrastructure will be introduced as part of the initial V1 architecture.

The retrieval flow is:

```text id="q4m8z1"
                    AI Task
                       │
                       ▼
              Deterministic Retrieval
                       │
            ┌──────────┴──────────┐
            │                     │
            ▼                     ▼
       Structured Data       Relationships
            │                     │
            └──────────┬──────────┘
                       ▼
                Context Builder
                       │
                       ▼
                AnalysisContext
                       │
                       ▼
                   AI Engine
```

Deterministic retrieval may use:

* project identifiers,
* entity types,
* explicit relationships,
* dates and time ranges,
* technologies,
* components,
* analysis identifiers,
* milestones,
* engineering events,
* engineering decisions,
* validated knowledge.

## Future Evolution

Semantic retrieval may be introduced when the system demonstrates a concrete need for it.

The future architecture may then evolve toward:

```text id="x3v7k2"
                     Core
                       │
              ┌────────┴────────┐
              │                 │
              ▼                 ▼
       Deterministic        Semantic
        Retrieval           Retrieval
              │                 │
              │            Vector Store
              │                 │
              └────────┬────────┘
                       ▼
                Context Ranking
                       │
                       ▼
                AnalysisContext
                       │
                       ▼
                   AI Engine
```

The introduction of semantic retrieval should be driven by observed requirements rather than assumptions.

Potential indicators for introducing semantic retrieval include:

* deterministic retrieval failing to find relevant historical knowledge,
* increasing project knowledge volume,
* repeated manual context selection,
* insufficient retrieval recall,
* difficulty discovering related knowledge across different terminology,
* AI tasks requiring broader historical context.

## Vector Store Ownership

When semantic retrieval is eventually introduced, the Vector Store will remain part of the Core's retrieval architecture.

The Core will remain responsible for:

* embedding lifecycle,
* vector indexing,
* semantic retrieval,
* combining deterministic and semantic results,
* context selection.

The AI Engine will continue to consume the resulting `AnalysisContext` and remain responsible for interpretation.

This preserves the architectural boundary established by ADR-014.

## Alternatives Considered

### Introduce semantic retrieval immediately

Add embeddings and a Vector Store during V1 development.

Rejected because:

* current knowledge volume does not justify the complexity,
* retrieval requirements are not yet validated,
* embedding strategy would be based on assumptions,
* infrastructure would increase operational complexity,
* the system can initially operate effectively with deterministic retrieval.

### Use the AI Engine as the initial Vector Store owner

Allow Python to generate embeddings and manage semantic retrieval.

Rejected because:

* it would move knowledge retrieval responsibility into the AI Engine,
* it would create a secondary representation of project knowledge,
* synchronization would become necessary,
* it would weaken the Core's role as the source of truth.

### Use an external managed Vector Database immediately

Introduce a managed vector database service from the beginning.

Rejected because:

* it adds external infrastructure before the need is demonstrated,
* it introduces additional operational and financial dependencies,
* the architecture can evolve toward this option later if justified.

### Never introduce semantic retrieval

Rely exclusively on deterministic retrieval indefinitely.

Rejected because:

* semantic relationships may become increasingly important as project knowledge grows,
* terminology may vary across historical knowledge,
* deterministic relationships may not capture all relevant conceptual connections.

## Consequences

### Positive

* Keeps V1 architecture simpler.
* Avoids premature infrastructure.
* Reduces operational complexity.
* Allows the team to learn from real project usage before selecting an embedding strategy.
* Keeps the Core architecture ready for future semantic retrieval.
* Avoids coupling the initial AI architecture to a specific Vector Store or embedding provider.
* Provides time to evaluate RAG, embeddings, vector databases, and hybrid retrieval properly.

### Negative

* V1 may have lower retrieval recall for semantically related historical knowledge.
* Some relevant knowledge may not be discovered through deterministic relationships.
* Semantic retrieval will require additional implementation when introduced.
* The retrieval layer may need to evolve as project knowledge grows.
