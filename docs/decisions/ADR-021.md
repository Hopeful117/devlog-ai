# ADR-021 - Project Source Model

## Status

Accepted

## Date

2026-07-21

## Context

DevLog AI analyzes the evolution, architecture, documentation, and engineering knowledge of software projects.

A software project may receive useful technical information from several origins:

* Git repositories;
* local directories;
* architecture documents;
* ADR directories;
* project documentation;
* issue trackers;
* pull requests;
* CI/CD systems;
* wikis;
* external APIs;
* archives.

The first implementation primarily needs access to Git repositories. However, modeling a repository directly as the only technical input of a project would make the domain unnecessarily dependent on Git.

DevLog AI is intended to understand software projects rather than merely analyze Git repositories.

The domain therefore needs a general concept representing an origin from which project information can be collected and analyzed.

At the same time, the V1 must avoid implementing a large generic ingestion framework before concrete use cases exist.

## Decision

DevLog AI will introduce `Source` as the domain concept representing an origin of analyzable project information.

A `Project` may own multiple Sources.



`Project` remains the primary business aggregate.

A Source is a technical input attached to that project.

The initial V1 will support only one concrete source type:

```text

```

Other source types may be added later without changing the fundamental relationship between Project and Source.

## Source Responsibilities

A Source answers:

> From where can DevLog AI collect evidence about this project?

A Source is responsible for representing:

* its type;
* its project ownership;
* its location or connection information;
* its activation state;
* source-specific configuration;
* synchronization metadata;
* creation and update timestamps.

Conceptually:


The exact entity fields may be refined during implementation, but the distinction between general source metadata and type-specific configuration must remain explicit.

## V1 Git Repository Source

The first supported Source type is a Git repository.

A Git source represents a repository associated with a project.

Its initial configuration may include:

```text
GitRepositorySource
в”њв”Ђв”Ђ repositoryUrl
в”њв”Ђв”Ђ defaultBranch
в”њв”Ђв”Ђ provider
в””в”Ђв”Ђ visibility
```

Only fields required by current use cases should be mandatory.

The V1 should not attempt to support every Git provider feature.

Initial providers may be represented using values such as:

```text
GITHUB
GITLAB
BITBUCKET
AZURE_DEVOPS
GENERIC_GIT
```

Provider-specific API integrations are outside the initial Source model scope.

A repository URL must be treated as source configuration rather than a direct field of `Project`.

## Project and Source Boundary

`Project` describes the software product or engineering initiative.

`Source` describes an origin from which technical evidence is collected.

For example:

```text
Project: Trading OS

Sources:
- trading-core Git repository
- broker-service Git repository
- market-data Git repository
- ai-engine Git repository
- frontend Git repository
```

This allows one logical project to span several repositories.

A single repository should not require creating a separate Project unless it represents a distinct product or engineering initiative.

## Analysis Scope

An `Analysis` continues to belong to a `Project`.

A future analysis may target:

* the complete project;
* one Source;
* a selected subset of Sources.

For the first increment, Analysis does not need to change unless source targeting is required immediately.

The future conceptual model is:

```text
Project
   в”‚
   в”њв”Ђв”Ђ Sources
   в”‚
   в””в”Ђв”Ђ Analysis
          в”‚
          в””в”Ђв”Ђ optional Source scope
```

The Source model must not prematurely require every Analysis to reference exactly one Source.

Project-level analyses must remain possible.

## Type Model

The Source type must be explicit.

Initial value:

```text
GIT_REPOSITORY
```

Potential future values include:

```text
LOCAL_DIRECTORY
DOCUMENTATION
ADR_DIRECTORY
ISSUE_TRACKER
CI_CD
WIKI
API
ARCHIVE
```

These future values document the intended extensibility of the domain but are not implementation requirements for V1.

The implementation should not create empty adapters, entities, or services for unsupported source types.

## Persistence Strategy

The V1 will use a common `Source` entity with an explicit `SourceType`.

Git-specific configuration may initially be represented using dedicated relational fields where the fields are stable and required.

Arbitrary unvalidated JSON configuration should be avoided unless a concrete need is demonstrated.

The V1 should favor an explicit and simple model over a generic source framework.

A future refactoring may introduce typed source configurations if additional source types demonstrate materially different requirements.

## Lifecycle

A Source may be active or inactive.

An inactive Source remains part of project history but is excluded from new collection or synchronization operations.

Deleting a Source should not silently remove historical analyses, facts, observations, or knowledge derived from it.

The exact archival and deletion policy may be refined later.

## Initial Implementation Scope

The first implementation increment includes:

* `Source` entity;
* `SourceType` with `GIT_REPOSITORY`;
* optional `GitProvider`;
* relation from Source to Project;
* database migration;
* repository;
* request and response DTOs;
* mapper;
* service and unit tests;
* controller;
* README;
* creation and retrieval by Project;
* activation or deactivation if consistent with existing project patterns.

The first increment does not include:

* Git cloning;
* GitHub API integration;
* repository authentication;
* webhook ingestion;
* automatic synchronization;
* local directory support;
* document ingestion;
* issue tracker integration;
* CI/CD integration;
* Analysis scoping by Source;
* deletion of derived historical knowledge.

## Alternatives Considered

### Store repository fields directly on Project

Add fields such as `repositoryUrl` and `defaultBranch` directly to `Project`.

Rejected because:

* a Project may span several repositories;
* Project would become coupled to Git;
* future non-Git inputs would require unrelated Project fields;
* repository lifecycle and configuration are separate from project identity.

### Model only Repository

Introduce a `Repository` entity as the sole project input abstraction.

Rejected because:

* DevLog AI is intended to understand projects using multiple forms of technical evidence;
* documentation, wikis, CI/CD, and issue trackers are not repositories;
* the domain would require replacement or significant refactoring when additional inputs are added.

### Implement a fully generic Source framework immediately

Create adapters and configuration models for Git, files, documents, wikis, APIs, issue trackers, and CI/CD from the beginning.

Rejected because:

* only Git is required for the V1;
* unsupported abstractions would be speculative;
* implementation and testing complexity would increase significantly;
* the system should evolve from demonstrated requirements.

### Treat each repository as a Project

Create one Project for every repository.

Rejected because:

* one logical software product may span several repositories;
* knowledge and milestones often belong to the product rather than one repository;
* cross-repository analyses would become fragmented;
* Trading OS and similar systems would lose their unified project context.

## Consequences

### Positive

* Preserves Project as the business aggregate.
* Supports multi-repository projects naturally.
* Keeps the domain open to future non-Git evidence sources.
* Avoids coupling Project directly to repository technology.
* Allows the V1 to remain focused on Git.
* Provides a clear future boundary for ingestion and synchronization.
* Improves long-term evidence traceability.

### Negative

* Introduces an additional domain entity.
* Git information requires an extra relationship instead of direct Project fields.
* Source configuration may require future refactoring as new source types are added.
* Source lifecycle and historical references will require careful handling.
* Future Analysis scoping across multiple Sources will require additional design.
