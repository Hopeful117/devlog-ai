# ADR-006 - AI Proposal and Knowledge Promotion Workflow

## Status

Accepted

## Date

2026-07-21

## Context

DevLog AI uses artificial intelligence to interpret structured project context and propose new project knowledge.

The Java Core remains the source of truth for the project domain and trusted project knowledge.

The AI Engine, however, is an interpretative system and its output cannot be considered authoritative by default.

AI-generated interpretations may be incorrect, incomplete, or based on assumptions that cannot be reliably inferred from the available context.

Examples include:

* architectural interpretations,
* engineering decisions,
* project events,
* technical challenges,
* documentation content,
* recommendations.

Automatically persisting AI-generated output as project knowledge would allow potentially incorrect information to enter the trusted knowledge base.

This would conflict with the principles established in:

* ADR-003 - Hybrid AI Architecture
* ADR-004 - Human Validation Workflow

The system therefore needs a clear architectural boundary between AI-generated proposals and validated project knowledge.

## Decision

DevLog AI will represent AI-generated outputs as `ValidatableProposal` objects.

AI-generated proposals are not considered authoritative project knowledge.

The AI Engine must never directly create or modify trusted project knowledge.

The promotion workflow is:

```text
AI Interpretation

↓

ValidatableProposal

↓

Human Validation

↓

┌─────────────────┬─────────────────┐
│                 │                 │
▼                 ▼                 │
ACCEPTED       REJECTED             │
│                 │                 │
▼                 ▼                 │
Knowledge      End of              │
Promotion      Proposal Lifecycle  │
│                                   │
▼                                   │
Validated Project Knowledge         │
└───────────────────────────────────┘
```

### Proposal Lifecycle

AI-generated proposals follow a controlled lifecycle.

```text
PROPOSED
   │
   ├── ACCEPTED
   │
   └── REJECTED
```

Once a proposal has been accepted or rejected, it becomes immutable.

A decided proposal cannot be modified or reused to represent a different interpretation.

If new analysis produces a different interpretation, a new proposal must be created.

For example:

```text
Analysis #1
    │
    └── Proposal #1 → REJECTED

Analysis #2
    │
    └── Proposal #2 → ACCEPTED
                         │
                         ▼
                  Project Knowledge
```

This preserves the history of AI-generated interpretations and human decisions.

### Knowledge Promotion

When a user accepts a proposal, the proposal validation and creation of the corresponding domain knowledge must occur within a single atomic transaction.

Conceptually:

```text
Accept Proposal
      │
      ▼
Begin Transaction
      │
      ├── Validate Proposal
      │
      ├── Create Domain Knowledge
      │
      └── Mark Proposal as ACCEPTED
      │
      ▼
Commit
```

If any operation fails, the transaction is rolled back.

The system must not reach a state where:

* a proposal is marked as accepted but the corresponding knowledge was not created;
* project knowledge exists without a corresponding accepted proposal.

The Java Core remains responsible for performing this promotion and enforcing domain consistency.

## Proposal Types

The `ValidatableProposal` concept may be used for different AI-generated outputs, including:

* Insight proposals,
* Engineering decision proposals,
* Engineering event proposals,
* Challenge proposals,
* Documentation proposals,
* Recommendations.

The proposal represents the AI-generated interpretation.

The resulting domain object represents validated project knowledge.

For example:

```text
InsightProposal
      │
      ▼
Human Validation
      │
      ▼
ACCEPTED
      │
      ▼
Insight
```

Or:

```text
DecisionProposal
      │
      ▼
Human Validation
      │
      ▼
ACCEPTED
      │
      ▼
Engineering Decision
```

The proposal and the resulting domain object are therefore distinct concepts.

## Responsibility Boundaries

### Java Core

The Java Core is responsible for:

* owning the `ValidatableProposal` lifecycle;
* storing AI-generated proposals;
* managing human validation;
* enforcing proposal state transitions;
* validating accepted proposals against domain rules;
* promoting accepted proposals into trusted project knowledge;
* maintaining transactional consistency;
* preserving proposal history.

### AI Engine

The AI Engine is responsible for:

* interpreting the `AnalysisContext`;
* generating AI proposals;
* producing structured proposal content;
* respecting the contracts defined by the Core.

The AI Engine does not:

* directly persist project knowledge;
* directly modify domain entities;
* decide whether a proposal is accepted;
* bypass human validation.

### Human Developer

The developer is responsible for:

* reviewing AI-generated proposals;
* accepting valid proposals;
* rejecting incorrect proposals;
* providing validation context when necessary.

## Alternatives Considered

### AI directly creates project knowledge

Allow the AI Engine to directly create or update domain objects.

Rejected because:

* AI output is not authoritative;
* incorrect interpretations could enter the trusted knowledge base;
* incorrect information could propagate into future analyses;
* human ownership of project knowledge would be weakened.

### Add validation status directly to domain entities

Add states such as `PROPOSED`, `ACCEPTED`, and `REJECTED` directly to entities such as `Insight` or `Decision`.

Rejected because:

* it mixes AI proposals with trusted domain knowledge;
* it makes validation a responsibility of each domain object;
* it creates unnecessary coupling between the AI workflow and the domain model;
* it becomes difficult to maintain a clear distinction between AI interpretation and validated knowledge.

### Generic validation attached to arbitrary domain objects

Create a generic validation mechanism based on a target entity and target identifier.

Rejected because:

* database referential integrity becomes difficult to enforce;
* polymorphic references weaken foreign key constraints;
* validation becomes detached from the AI proposal lifecycle;
* the model does not clearly represent the distinction between proposed and trusted knowledge.

### Mutable proposals

Allow rejected or accepted proposals to be modified and reused.

Rejected because:

* the history of AI interpretations could be lost;
* previous human decisions could become ambiguous;
* auditability and traceability would be reduced.

### Manual recreation after acceptance

Require the user or application to manually recreate the domain object after accepting a proposal.

Rejected because:

* acceptance and knowledge creation could become inconsistent;
* accepted proposals could remain without corresponding knowledge;
* additional user interaction would introduce unnecessary friction.

## Consequences

### Positive

* AI output is clearly separated from trusted project knowledge.
* The Java Core remains the authoritative source of truth.
* Human validation is enforced before AI-generated knowledge becomes official.
* AI interpretations remain traceable.
* Historical proposals and decisions are preserved.
* Accepted knowledge is created atomically.
* The architecture supports multiple types of AI-generated proposals.
* The AI Engine remains replaceable and isolated from the domain model.

### Negative

* AI-generated knowledge cannot be fully autonomous.
* Human interaction is required for promotion of AI proposals.
* The system requires additional persistence and lifecycle management for proposals.
* Proposal-to-domain promotion introduces additional application logic.
* The architecture is more complex than directly persisting AI output.
