# ADR-012 - Core as the Source of Truth for AI Workflow State

## Status

Accepted

## Date

2026-07-21

## Context

DevLog AI uses asynchronous AI processing between the Java Core and the Python AI Engine.

The Core orchestrates the business workflow, while the AI Engine performs the internal execution of AI tasks.

Both services may therefore need to maintain state related to AI processing.

The architecture must distinguish between:

* business workflow state,
* internal AI execution state.

Without a clear distinction, both services could attempt to maintain an authoritative representation of the same workflow.

This would create synchronization problems and make it difficult to determine which state is authoritative when failures or communication errors occur.

## Decision

The Java Core is the single source of truth for the business state of AI workflows.

The Python AI Engine may maintain internal execution state required to process AI tasks, but this state is not authoritative for the DevLog AI business workflow.

The responsibility boundary is:

```text id="z7m4q1"
                    JAVA CORE
                Source of Truth
                       │
                       │ AI Task
                       ▼
              PYTHON AI ENGINE
              Internal Execution
                       │
                       ├── Processing
                       ├── Model Selection
                       ├── Inference
                       └── Internal Job State
                       │
                       ▼
                 AI Task Result
                       │
                       ▼
                    JAVA CORE
                       │
                       ▼
             Business Workflow State
```

The Core owns the lifecycle of business-level AI tasks.

For example:

```text id="b4k8n2"
AI Task
    │
    ▼
CREATED
    │
    ▼
SUBMITTED
    │
    ▼
PROCESSING
    │
    ▼
COMPLETED
    │
    ▼
PROPOSALS_CREATED
    │
    ▼
WAITING_VALIDATION
    │
    ├── ACCEPTED
    │
    └── REJECTED
```

The exact state machine may evolve as the system develops.

The AI Engine may maintain a separate internal execution lifecycle:

```text id="p9x3v7"
AI Job
    │
    ▼
RECEIVED
    │
    ▼
PROCESSING
    │
    ▼
INFERENCE
    │
    ▼
COMPLETED
```

The AI Engine's internal lifecycle is an implementation detail and must not become a second authoritative business state machine.

## Source of Truth Principle

The following rule applies:

> The Core owns what the AI workflow means to the DevLog AI domain. The AI Engine owns only how an individual AI task is executed.

The Core therefore remains authoritative for:

* whether an AI task exists,
* why it exists,
* which analysis it belongs to,
* which context was provided,
* whether processing was requested,
* whether a result was received,
* whether proposals were created,
* whether proposals are awaiting validation,
* whether proposals were accepted or rejected.

The AI Engine is authoritative only for its own internal processing state.

## Correlation

Each AI task exchanged between services must be traceable through a stable identifier.

Conceptually:

```text id="e3k6p8"
Analysis
    │
    └── AI Task ID
             │
             ├── Core AI Task
             │
             └── AI Engine Internal Job
```

The correlation identifier allows the Core to associate asynchronous AI results with the correct business workflow.

The exact implementation of identifiers and correlation metadata may evolve independently from this architectural decision.

## Failure Handling

If communication between services fails, the Core remains authoritative.

For example:

```text id="m8v2q4"
Core
  │
  │ Submit AI Task
  ▼
AI Engine
  │
  X Processing failure
  │
  ▼
Core
```

The Core determines the business state resulting from the failure.

The AI Engine may report technical failure information, but it does not decide the business consequences of that failure.

Possible business states may include:

```text id="x4n7q2"
FAILED
RETRY_PENDING
CANCELLED
```

These states belong to the Core's workflow model.

## Alternatives Considered

### AI Engine as the source of truth

Allow the Python service to own the authoritative AI task lifecycle.

Rejected because:

* AI processing would become coupled to business workflow state,
* the Core would need to synchronize its state with Python,
* project knowledge and workflow ownership would become distributed,
* failure recovery would become more complex.

### Both services maintain authoritative state

Allow both services to independently maintain the same AI workflow state.

Rejected because:

* competing sources of truth would exist,
* state synchronization would be required,
* inconsistent states could occur,
* debugging and recovery would become more difficult.

### No persistent workflow state

Treat AI processing as a stateless request-response operation.

Rejected because:

* AI processing is asynchronous,
* processing may take significant time,
* failures and retries require state,
* human validation requires persistent workflow information,
* historical traceability is important to DevLog AI.

## Consequences

### Positive

* Establishes a single authoritative source of truth.
* Simplifies failure recovery.
* Keeps business state in the Core.
* Allows the AI Engine to evolve independently.
* Supports asynchronous processing.
* Improves traceability and correlation.
* Prevents AI implementation details from leaking into the domain model.

### Negative

* The Core must persist and manage AI workflow state.
* Correlation identifiers are required.
* The Core must handle asynchronous failure and retry semantics.
* The AI Engine's internal state may not always map one-to-one with Core workflow states.
