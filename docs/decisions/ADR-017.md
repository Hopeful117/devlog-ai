# ADR-017 - AI Task Domain and Lifecycle

## Status

Accepted

## Date

2026-07-21

## Context

DevLog AI delegates AI interpretation to a separate Python AI Engine.

The Java Core remains responsible for the overall business workflow and is the authoritative source of truth for AI task state.

AI processing is asynchronous and uses REST communication between the Core and the AI Engine.

The system therefore needs a persistent domain model representing an AI task submitted by the Core.

This model must support:

* correlation between the Core and AI Engine;
* association with the originating analysis;
* traceability of the exact context sent to the AI Engine;
* asynchronous processing states;
* technical failure information;
* future retries;
* multiple AI tasks for a single analysis.

The AI task model must remain independent from the internal implementation of the Python AI Engine.

## Decision

DevLog AI will introduce a persistent `AiTask` domain owned by the Java Core.

An `AiTask` represents one AI processing request initiated by the Core for a specific `Analysis`.

The relationship is:

### `id`

The internal identifier of the AI task in the Core.

### `analysis`

The analysis that produced the context and initiated the AI task.

An AI task must always belong to one analysis.

### `correlationId`

A stable and unique identifier generated by the Core.

The correlation identifier is used to match:

* the Core task;
* the request sent to the AI Engine;
* the AI Engine internal job;
* the asynchronous callback result.

The AI Engine must not generate or replace the Core correlation identifier.

### `taskType`

The type of AI work requested.

`taskType` is distinct from `AnalysisType`.

An `AnalysisType` describes the analytical purpose of an analysis.

An `AiTaskType` describes one specific AI operation performed within that analysis.

For example:

```text
AnalysisType:
ARCHITECTURE_REVIEW

AiTaskType:
DECISION_PROPOSAL_GENERATION
DOCUMENTATION_GENERATION
INSIGHT_GENERATION
```

This distinction allows one analysis to execute multiple AI tasks.

The statuses mean:

* `CREATED`: the task exists in the Core but has not yet been submitted;
* `SUBMITTED`: the Core has submitted the task to the AI Engine;
* `PROCESSING`: the AI Engine has started processing the task;
* `COMPLETED`: the AI Engine completed the task successfully;
* `FAILED`: the task ended with a technical or processing failure.

The exact transition mechanism will be implemented in the Core service layer.

### `contextSnapshot`

The Core will persist the exact context sent to the AI Engine.

The snapshot is immutable after task creation.

It will initially be stored as PostgreSQL `JSONB`.

The snapshot provides:

* reproducibility;
* debugging;
* historical traceability;
* AI output evaluation;
* auditability.

The task must not depend on rebuilding the context later to explain what the AI Engine received.

### `externalJobId`

An optional identifier returned by the AI Engine.

It may represent an internal Python job or provider-specific execution identifier.

It is not authoritative for the Core workflow.

### `attemptCount`

The number of submission or processing attempts.

The initial value is zero or one according to the implementation convention selected by the service.

Automatic retries are not part of the first implementation increment.

The field is included to preserve future extensibility.

### Failure fields

The Core may store:

```text
failureCode
failureMessage
```

These fields represent technical failure information reported by the AI Engine or detected by the Core.

Failure metadata does not replace the Core task status.

### Timestamps

The task records lifecycle timestamps:

```text
createdAt
submittedAt
startedAt
completedAt
```

These timestamps provide execution traceability and allow the system to measure queue time and processing duration.

## Lifecycle Ownership

The Java Core is responsible for:

* creating the task;
* generating the correlation identifier;
* assigning the initial status;
* persisting the context snapshot;
* validating status transitions;
* storing external execution identifiers;
* recording failures;
* determining the final business state.

The Python AI Engine may maintain its own internal execution lifecycle, but it does not own the Core `AiTask` state.

The ownership boundary is:

## Immutability

The following fields are immutable after task creation:

* `analysis`;
* `correlationId`;
* `taskType`;
* `contextSnapshot`;
* `createdAt`.

Lifecycle fields may only change through explicit service-level state transitions.

Direct arbitrary updates to AI task state are not allowed.

## Initial Implementation Scope

The first implementation increment will include:

* entity;
* migration;
* repository;
* DTOs;
* mapper;
* service interface;
* service implementation;
* unit tests;
* controller;
* README.

The first increment will not include:

* REST communication with the AI Engine;
* asynchronous callback handling;
* automatic retries;
* task cancellation;
* proposal creation from AI results;
* transition of `Analysis` based on task state;
* automatic knowledge promotion.

These capabilities will be implemented in later increments.

## Alternatives Considered

### Use `Analysis` directly as the AI task

Rejected because:

* one analysis may require multiple AI operations;
* AI execution state would be mixed with deterministic analysis state;
* correlation with individual AI operations would be impossible;
* retries and failures would be harder to isolate.

### Let the AI Engine own AI task persistence

Rejected because:

* the Core would no longer be the source of truth;
* business workflow state would become distributed;
* correlation and recovery would become more complex;
* task history would depend on Python infrastructure.

### Do not persist the context snapshot

Rejected because:

* the exact AI input could not be reproduced;
* debugging would depend on rebuilding context;
* historical analysis could change after the task was executed;
* AI output evaluation would be less reliable.

### Use only `AnalysisType`

Rejected because:

* one analysis can require multiple AI tasks;
* analysis purpose and AI operation are different concepts;
* task specialization would become difficult.

### Add workflow states such as `PROPOSALS_CREATED` and `WAITING_VALIDATION`

Rejected for the initial `AiTask` lifecycle because:

* these states describe downstream proposal and validation workflows;
* they are not part of the technical execution of an AI task;
* mixing them would blur domain boundaries.

They may instead belong to the surrounding orchestration workflow.

## Consequences

### Positive

* Provides a persistent and traceable AI execution model.
* Keeps the Core as the authoritative source of truth.
* Supports multiple AI tasks per analysis.
* Preserves the exact AI input context.
* Enables asynchronous correlation and future callbacks.
* Supports future retries and observability.
* Keeps AI Engine implementation details outside the Core domain.

### Negative

* Introduces another persistent domain.
* Requires explicit lifecycle transition logic.
* Context snapshots may consume significant database storage.
* Future cleanup or retention policies may be required.
* The distinction between task state and surrounding workflow state must remain clear.
