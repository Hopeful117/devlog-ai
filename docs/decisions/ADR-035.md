# ADR-035 — Project History Understanding

* **Status:** Accepted
* **Date:** 2026-07-23

## Context

DevLog AI is designed to become a living technical memory capable of explaining the current state of a software project and reconstructing how that state was reached.

The repository already contains several sources of historical knowledge:

* Git commits;
* Architecture Decision Records;
* roadmap items;
* deterministic analyses;
* validated Insights;
* AI tasks and proposals;
* generated deliverables.

At present, these elements are primarily interpreted independently. This limits DevLog's ability to produce a faithful project timeline, relate implementation work to architectural decisions, or explain how a capability evolved across several commits.

Commit messages alone are insufficient for reconstructing project history. They may be incomplete, imprecise, inconsistent or focused only on the immediate implementation detail.

DevLog therefore requires an explicit historical model connecting Git activity to trusted project knowledge.

## Decision

DevLog AI will introduce a dedicated project-history understanding capability.

Project history will be reconstructed from chronological Git activity and its relationships with:

* ADRs;
* roadmap items;
* deterministic analyses;
* validated Insights;
* AI tasks;
* generated deliverables;
* existing trusted knowledge.

The objective is not merely to list commits, but to identify meaningful project evolutions and preserve them as traceable historical knowledge.

## Historical Sources

The history engine may consume:

* commit hashes;
* parent commit hashes;
* commit dates;
* authors;
* commit messages;
* changed files;
* branches and tags;
* ADR metadata;
* roadmap metadata;
* analysis results;
* validated knowledge.

Each source contributes context but does not independently define the historical truth.

## Historical Model

DevLog should distinguish three levels of information.

### Raw Git Activity

Represents factual Git information:

* commit;
* parent;
* author;
* timestamp;
* message;
* changed files;
* insertion and deletion statistics.

### Commit Interpretation

Represents an interpretation of the purpose and impact of one commit.

Examples include:

* capability introduced;
* defect corrected;
* architecture refactored;
* tests strengthened;
* documentation aligned.

AI-generated interpretations remain proposals until validated.

### Significant Project Event

Represents a meaningful evolution that may span one or several commits.

Examples include:

* introduction of Intent execution and User Guidance;
* completion of an ADR implementation;
* migration to a new architecture;
* delivery of a product milestone;
* removal of a deprecated approach.

Several small commits may be grouped into one significant historical event.

## Relationships

Historical elements should support explicit relationships such as:

* commit implements ADR;
* commit contributes to roadmap item;
* commit fixes analysis finding;
* commit introduces capability;
* commit completes milestone;
* commit supersedes previous implementation;
* event produces validated knowledge;
* deliverable documents event.

Relationships may initially be proposed by AI but must follow the existing validation lifecycle before becoming trusted knowledge.

## Proposal Lifecycle

In accordance with the existing proposal architecture:

* AI-generated historical interpretations are represented as immutable `ValidatableProposal` objects;
* proposals are never trusted project knowledge directly;
* human or deterministic validation is required;
* accepted proposals are promoted atomically into trusted historical knowledge;
* rejected proposals remain immutable and traceable;
* a new interpretation creates a new proposal.

## User Guidance

Historical analyses may be influenced by User Guidance.

Examples include:

* focus on architecture;
* focus on product capabilities;
* focus on testing and quality;
* focus on a particular milestone;
* requested level of detail;
* target audience.

User Guidance affects presentation and selection priorities but must not override factual Git data or trusted knowledge.

## Expected Capabilities

DevLog should progressively become capable of answering:

* What changed since the previous report?
* When was a capability introduced?
* Which commits implemented a specific ADR?
* How did a feature evolve over time?
* Which roadmap items are complete or still active?
* Why does the current architecture exist?
* Which significant project events occurred during a period?
* What was the last known planned development step?

## Project State Reports

Project history becomes a primary source for state reports.

A report should be able to compare:

* previous known state;
* new Git activity;
* newly validated knowledge;
* roadmap evolution;
* newly completed or incomplete capabilities.

This allows DevLog to produce incremental reports rather than repeatedly summarizing the repository from scratch.

## Traceability

Every historical interpretation should retain references to its evidence.

Evidence may include:

* commit hashes;
* changed files;
* ADR identifiers;
* roadmap item identifiers;
* analysis identifiers;
* validated Insight identifiers.

A historical claim should remain explainable and auditable.

## Consequences

### Advantages

* faithful reconstruction of project evolution;
* better state reports;
* improved continuity after long development pauses;
* stronger relationships between code, ADRs and roadmap;
* richer context for future AI agents;
* improved generation of technical articles and handover documentation.

### Trade-offs

* additional historical domain models;
* relationship management complexity;
* potential duplicate or conflicting interpretations;
* need for incremental re-analysis as knowledge evolves.

These trade-offs are accepted because project-history understanding is a core capability of DevLog AI.

## Alternatives Considered

### Use Commit Messages as Project History

Rejected.

Commit messages are useful context but are not sufficiently complete or reliable.

### Generate Reports From the Current Repository Only

Rejected.

A repository snapshot explains what currently exists but not how or why it evolved.

### Persist AI Interpretations Directly

Rejected.

Historical interpretations must respect the immutable proposal and human-validation lifecycle.
