# ADR-005 - Microservice Boundaries

## Status

Accepted

## Date

2026-07-20

## Context

DevLog AI combines multiple technical domains with different responsibilities and evolution cycles.

The system requires:

- a reliable business domain handling project knowledge,
- specialized artificial intelligence capabilities,
- a user interface for interaction and validation.

These domains have different technical constraints and should evolve independently.

A single application containing all responsibilities would create strong coupling between business logic, AI experimentation, and user interface concerns.

However, excessive service decomposition would introduce unnecessary complexity and reduce development velocity.

## Decision

DevLog AI will use a microservice-oriented architecture based on clear responsibility boundaries.

The initial architecture will contain three main application components:

- Angular Frontend
- Java Spring Boot Core Service
- Python AI Service

An API Gateway will provide a unified entry point for frontend communication.

The PostgreSQL database will store the Core Service domain data.

## Service Responsibilities

### Java Core Service

The Core Service is the source of truth for project knowledge.

Responsibilities:

- domain logic,
- project management,
- repository management,
- knowledge lifecycle management,
- validation workflow,
- persistence.

The Core Service owns business rules and validated project knowledge.

### Python AI Service

The AI Service provides specialized intelligence capabilities.

Responsibilities:

- semantic analysis,
- AI model interaction,
- knowledge interpretation,
- proposal generation,
- content generation.

The AI Service does not own project knowledge and only works with contextual data provided by the Core Service.

### Angular Frontend

The frontend provides user interaction with the platform.

Responsibilities:

- project visualization,
- knowledge exploration,
- proposal validation,
- documentation workflows.

## Alternatives Considered

### Modular monolith

Keep all backend responsibilities inside a single Spring application.

Rejected because:

- AI capabilities have different technical requirements,
- Python provides a stronger ecosystem for AI development,
- service boundaries already exist naturally.

### Highly fragmented microservices

Create many specialized services from the beginning.

Rejected because:

- increases operational complexity,
- slows development,
- creates unnecessary communication overhead.

## Consequences

### Positive

- Clear separation of responsibilities.
- Independent evolution of AI capabilities.
- Technology choices can match domain requirements.
- Architecture demonstrates real-world distributed system practices.

### Negative

- Requires service communication.
- Increases deployment complexity.
- Requires stronger infrastructure and monitoring practices.
