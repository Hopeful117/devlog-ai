# ADR-039 — Context Intelligence

* **Status:** Accepted
* **Date:** 2026-07-23

## Context

ADR-037 establishes the **Repository-First** strategy.

ADR-038 introduces the **Repository Context Engine**, responsible for collecting, ranking and assembling repository evidence into a structured `AnalysisContext`.

However, not every AI analysis requires the same repository perspective.

For example:

* an architecture review should prioritize ADRs and architectural relationships;
* a project state report should prioritize recent commits and roadmap progress;
* README generation should prioritize implemented capabilities and project documentation.

Selecting repository evidence therefore requires an additional decision layer responsible for choosing the most appropriate context strategy for a given analysis.

## Decision

DevLog AI introduces a **Context Intelligence** layer.

Context Intelligence determines **how repository context should be constructed** for a specific analysis.

It is responsible for selecting context strategies, context profiles and evidence ranking policies before the Repository Context Engine builds the final `AnalysisContext`.

The Context Intelligence layer remains deterministic.

Artificial Intelligence does not participate in repository context selection.

---

## Architecture

The analysis pipeline becomes:

```text
Intent
      │
      ▼
Context Intelligence
      │
      ▼
Context Profiles
      │
      ▼
Repository Context Engine
      │
      ▼
AnalysisContext
      │
      ▼
AI Engine
```

Each layer owns a distinct responsibility.

---

## Responsibilities

Context Intelligence is responsible for:

* selecting context profiles;
* composing multiple context profiles;
* selecting ranking strategies;
* selecting evidence weighting policies;
* determining evidence diversity requirements;
* enforcing explainable evidence selection.

It is not responsible for collecting repository information.

Repository collection remains the responsibility of ADR-038.

---

## Context Profiles

Context Profiles represent reusable repository perspectives.

Examples include:

* Project State;
* Architecture;
* History;
* Documentation;
* Release.

Profiles are reusable across multiple Intents.

An Intent references one or more Context Profiles rather than implementing its own selection logic.

---

## Context Profile Composition

Multiple Context Profiles may be combined.

Examples include:

```
describe-project-v1

↓

Project State
+
History
```

```
architecture-overview-v1

↓

Architecture
+
History
```

Composition remains deterministic.

---

## Evidence Scoring

Repository evidence should no longer rely on a single arbitrary relevance score.

Instead, evidence is evaluated using multiple independent criteria.

Typical criteria include:

* semantic relevance;
* architectural relevance;
* historical relevance;
* recency;
* confidence;
* User Guidance boost.

These criteria produce a calculated final relevance score.

The final score is derived from the active Context Profile rather than directly assigned.

---

## Context Strategies

Each Context Profile defines an internal strategy describing:

* evidence priorities;
* weighting policies;
* recency policies;
* diversity policies;
* token allocation priorities.

Strategies remain internal implementation details.

---

## User Guidance

User Guidance may influence evidence prioritization.

Examples include:

* focusing on architecture;
* emphasizing testing;
* reducing implementation details.

User Guidance provides additional weighting but never replaces the Context Profile.

Repository evidence selection therefore remains deterministic and reproducible.

---

## Configuration

Context Profiles are:

* predefined;
* versioned;
* deterministic;
* reusable.

They are **not** user configurable.

This avoids excessive configuration complexity while preserving predictable behaviour.

Future profile improvements should occur through versioned implementations rather than runtime configuration.

---

## Explainability

The Context Intelligence layer should be capable of explaining:

* why a Context Profile was selected;
* why evidence was ranked highly;
* why evidence was discarded;
* how the final AnalysisContext was assembled.

This explainability supports future debugging, validation and trust.

---

## Consequences

### Advantages

* reusable context strategies;
* deterministic evidence selection;
* explainable ranking;
* improved prompt quality;
* easier Intent creation;
* reduced duplication between analyses.

### Trade-offs

* additional decision layer;
* more sophisticated evidence scoring;
* increased implementation complexity.

These trade-offs are accepted because context quality directly influences analysis quality.

---

## Alternatives Considered

### Each Intent Defines Its Own Context Logic

Rejected.

This would duplicate selection logic and make future maintenance difficult.

---

### Fully Configurable Context Profiles

Rejected.

Highly configurable profiles would significantly increase complexity and reduce predictability.

Versioned predefined profiles provide a better balance between flexibility and maintainability.

---

### Single Relevance Score

Rejected.

A single arbitrary score cannot explain why evidence was selected.

Multi-criteria scoring provides better transparency and allows profile-specific weighting.
