# ADR-034 — Deliverable Generation Pipeline

## Status

Accepted

## Date

2026-07-22

---

# Context

DevLog AI currently implements the complete engineering knowledge pipeline.

```text
Repository
        │
        ▼
Collectors
        │
        ▼
Facts
        │
        ▼
Observation Engine
        │
        ▼
Observations
        │
        ▼
Project Profile
        │
        ▼
Knowledge Selection
        │
        ▼
Prompt Builder
        │
        ▼
LLM Provider
        │
        ▼
InsightProposal
        │
        ▼
Human Validation
        │
        ▼
Insight
```

This pipeline produces validated engineering knowledge.

However, the original business objective of DevLog AI is not merely to store validated knowledge.

The objective is to help engineers produce useful engineering artifacts.

Examples include:

- project descriptions;
- README files;
- architecture summaries;
- portfolio pages;
- technical documentation;
- technical articles;
- future implementation tasks.

The current pipeline therefore stops one step too early.

---

# Problem

Insights represent validated engineering knowledge.

They are not the final deliverable expected by the user.

A user asking:

> Describe this project

does not expect:

- four validated Insights

The user expects:

- one coherent project description.

The same applies to:

- documentation;
- articles;
- summaries;
- reports.

A second transformation stage is therefore required.

---

# Decision

DevLog AI introduces a dedicated Deliverable Generation Pipeline.

Validated Insights become the only business input accepted for content generation.

```text
Repository
        │
        ▼
Knowledge Pipeline
        │
        ▼
Validated Insights
        │
        ▼
Deliverable Generation
        │
        ▼
Generated Deliverable
```

---

# Architectural Principle

Knowledge is produced first.

Knowledge is validated second.

Only validated knowledge may produce deliverables.

No deliverable may be generated directly from:

- repository source code;
- Facts;
- Observations;
- AnalysisContext;
- Prompt outputs that were not validated.

---

# Deliverable Generator

A new component named Deliverable Generator is introduced.

Responsibilities:

- generate user-facing artifacts;
- organize validated knowledge;
- adapt content to the requested audience;
- preserve engineering correctness;
- remain independent from repository analysis.

Non responsibilities:

- repository analysis;
- Fact collection;
- Observation generation;
- InsightProposal generation;
- Human Validation.

---

# Input

The Deliverable Generator receives:

```text
DeliverableRequest

├── Project
├── Analysis (optional)
├── Deliverable Type
├── Audience
├── Style
├── Language
├── Validated Insights
└── Additional User Guidance
```

The Deliverable Generator never receives repository data directly.

---

# Deliverable Types

The initial supported types include:

- PROJECT_DESCRIPTION
- README
- ARCHITECTURE_SUMMARY
- PORTFOLIO_DESCRIPTION
- TECHNICAL_SUMMARY
- BLOG_ARTICLE

Future types may include:

- IMPLEMENTATION_TASKS
- RELEASE_NOTES
- MIGRATION_GUIDES
- ONBOARDING_GUIDES

---

# Human Validation Boundary

Human validation occurs before deliverable generation.

The Deliverable Generator only consumes:

```text
Insight
```

never:

```text
InsightProposal
```

This guarantees that generated documents rely exclusively on trusted engineering knowledge.

---

# AI Responsibility

The Deliverable Generator may internally use an LLM.

Its responsibility is fundamentally different from the Analysis AI.

Analysis AI answers:

"What knowledge exists?"

Deliverable AI answers:

"How should this validated knowledge be communicated?"

The Deliverable Generator is therefore a communication engine, not a reasoning engine.

---

# Deterministic Knowledge Boundary

The Deliverable Generator must not invent new engineering knowledge.

It may:

- reorder;
- summarize;
- simplify;
- explain;
- adapt tone;
- adapt audience.

It must not:

- introduce unsupported technical claims;
- infer undocumented architecture;
- create additional Insights.

---

# Traceability

Every generated deliverable stores:

- Deliverable Type
- Generation Time
- Language
- Audience
- Style
- Deliverable Prompt Version
- Deliverable Prompt Digest
- Source Insight IDs

This allows complete reproducibility.

---

# Regeneration

Deliverables are reproducible.

Given:

- identical validated Insights;
- identical Deliverable Type;
- identical Prompt Version;
- identical User Guidance;

DevLog AI must be capable of regenerating an equivalent deliverable.

---

# Future Architecture

```text
Validated Insights
        │
        ▼
Deliverable Selection
        │
        ▼
Deliverable Prompt Builder
        │
        ▼
LLM Provider
        │
        ▼
Generated Deliverable
```

This pipeline intentionally mirrors the Analysis Pipeline.

---

# Future Integration

Future consumers may include:

- Portfolio
- CodeGlyphe
- Trading OS documentation
- DevLog AI itself

The Deliverable Generator becomes a shared engineering communication service.

---

# Consequences

## Positive

- Complete engineering workflow.
- Clear separation between reasoning and communication.
- Strong Human-in-the-Loop guarantee.
- Highly reusable validated knowledge.
- Multiple deliverables generated from the same Insights.
- Better long-term maintainability.

## Negative

- Additional pipeline.
- Additional Prompt Templates.
- Additional persistence layer.

---

# Alternatives Considered

## Generate documents directly from AnalysisContext

Rejected.

Would bypass validated engineering knowledge.

## Generate documents directly from InsightProposal

Rejected.

Would bypass Human Validation.

## Store generated documents manually

Rejected.

Does not support reproducibility.

---

# Architectural Principle

DevLog AI distinguishes two fundamentally different AI responsibilities.

Analysis AI discovers engineering knowledge.

Humans validate engineering knowledge.

Deliverable AI communicates validated engineering knowledge.

Only validated knowledge may become engineering communication.
