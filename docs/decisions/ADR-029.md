# ADR-029 — Insight Domain Model

## Status

Accepted

## Date

2026-07-22

---

# Context

Previous ADRs progressively established the deterministic knowledge pipeline.

```text
Repository
    ↓
Workspace
    ↓
Collectors
    ↓
Facts
    ↓
Observations
    ↓
Project Profile
    ↓
AnalysisContext
    ↓
Intent
```

At this point, DevLog AI possesses a deterministic understanding of a software project before invoking any Large Language Model.

The AI Engine never operates directly on repository files.

Instead, it consumes an immutable `AnalysisContext` and produces one or more `InsightProposal` objects.

These proposals are reviewed by a human before becoming trusted project knowledge.

Although previous ADRs defined the lifecycle of proposals, no architectural decision formally defines the Insight domain itself.

This ADR establishes Insight as the primary business object produced by DevLog AI.

---

# Decision

DevLog AI defines **Insight** as the immutable, human-validated semantic interpretation of deterministic project knowledge produced in response to an explicit user Intent.

An Insight is **not** an AI response.

An Insight is the validated business knowledge obtained after reviewing an AI-generated InsightProposal.

---

# Knowledge Pipeline

The complete knowledge pipeline becomes:

```text
Evidence
    ↓
Facts
    ↓
Observations
    ↓
Project Profile
    ↓
AnalysisContext
    ↓
Intent
    ↓
InsightProposal
    ↓
Human Validation
    ↓
Insight
```

Each layer increases the semantic level of the knowledge.

---

# Business Definition

An Insight answers:

> **What did this Analysis understand about the Project?**

It does **not** answer:

- What files exist?
- Which Facts were collected?
- Which Observations were detected?

Those questions belong to lower deterministic layers.

An Insight represents an interpretation of deterministic knowledge.

---

# Human Validation

InsightProposal and Insight are distinct concepts.

```text
InsightProposal
        │
        ├── Accepted
        │         │
        │         ▼
        │      Insight
        │
        └── Rejected
```

The AI Engine can never create an Insight directly.

Only a human validation transforms an InsightProposal into trusted project knowledge.

Human validation remains mandatory regardless of AI provider quality.

---

# Relationship with Intent

An Insight is always generated in response to a specific Intent.

Intent defines:

- why the AI is invoked;
- expected output;
- allowed Insight categories.

However, an Insight remains independent from the Intent after validation.

The same validated Insight may later be reused by different Intents.

For example:

```text
Describe Project

↓

Architecture Insight
```

may later be reused for:

- README generation;
- Architecture documentation;
- Project presentation;
- Technical blog article.

The Intent defines the request.

The Insight represents reusable project knowledge.

---

# Relationship with Analysis

Every Insight belongs to:

- exactly one Project;
- exactly one Analysis.

```text
Project
      │
      ▼
Analysis
      │
      ├── Insight
      ├── Insight
      └── Insight
```

The Analysis remains the historical boundary of the generated knowledge.

---

# Relationship with AnalysisContext

Every Insight originates from one immutable AnalysisContext.

Therefore every Insight can be traced back to:

```text
Insight
    ↓
InsightProposal
    ↓
Intent
    ↓
AnalysisContext
    ↓
Project Profile
    ↓
Observations
    ↓
Facts
    ↓
Evidence
```

This complete provenance chain must remain preserved.

---

# Insight Types

Insight Types classify the business nature of the knowledge.

Initial taxonomy:

```text
ARCHITECTURAL

DOCUMENTATION

TECHNOLOGY

EVOLUTION

TECHNICAL_DEBT

SECURITY

RISK

RECOMMENDATION
```

Additional types may be introduced through future ADRs.

Changing an existing semantic meaning requires a new version.

---

# Insight Severity

Insights are classified by importance.

Initial levels:

```text
INFO

WARNING

CRITICAL
```

Severity expresses business importance.

It is **not** an AI confidence score.

---

# Insight Responsibilities

The Insight domain is responsible for:

- storing validated project knowledge;
- associating knowledge with Projects;
- associating knowledge with Analyses;
- preserving provenance;
- categorizing knowledge;
- assigning severity;
- enabling historical retrieval;
- enabling filtering by type and severity;
- enabling future document generation.

---

# Insight Non-Responsibilities

The Insight domain is not responsible for:

- executing AI models;
- prompt engineering;
- provider selection;
- repository analysis;
- knowledge collection;
- Observation generation;
- Project Profile generation;
- AnalysisContext construction;
- Human validation workflow.

Those responsibilities belong to other bounded contexts.

---

# Insight Immutability

A validated Insight is immutable.

Corrections never modify an existing Insight.

Instead:

```text
New Analysis
        ↓
New InsightProposal
        ↓
New Validation
        ↓
New Insight
```

Historical knowledge must remain reproducible.

---

# Insight Independence

Insights are independent from presentation.

An Insight is **not**:

- a README section;
- a documentation page;
- a blog article;
- a markdown paragraph.

Those artifacts are projections of validated Insights.

---

# Document Projection

User-visible documents become deterministic compositions of Insights.

Examples:

```text
Insights
      │
      ▼
README
```

```text
Insights
      │
      ▼
Architecture Documentation
```

```text
Insights
      │
      ▼
Project Presentation
```

```text
Insights
      │
      ▼
Blog Article
```

Multiple documents may reuse the same Insight.

No document owns the underlying knowledge.

---

# Architectural Principle

Facts describe what exists.

Observations describe what can be objectively concluded.

Project Profiles describe the project as a whole.

Insights describe what the project knowledge means in response to a user Intent.

Only validated Insights become trusted knowledge.

---

# Consequences

## Positive

- Clear business definition.
- Human validation remains central.
- Strong separation between AI responses and project knowledge.
- Reusable knowledge independent from documents.
- Stable historical traceability.
- Extensible taxonomy.
- Strong foundation for future documentation generation.

## Negative

- Additional business abstraction.
- Requires lifecycle management.
- Requires taxonomy maintenance.

---

# Alternatives Considered

## AI responses as project knowledge

Rejected.

AI output remains probabilistic.

Project knowledge must remain validated.

---

## Documents as primary artifacts

Rejected.

Documentation should be generated from reusable knowledge.

Knowledge should never depend on one document format.

---

## Insights coupled to Intent

Rejected.

Intent defines the request.

Insight represents reusable validated knowledge.

The same Insight may satisfy multiple future Intents.

---

# Architectural Vision

DevLog AI does not generate documentation.

DevLog AI generates validated project knowledge.

Documentation, README files, technical articles, architecture descriptions and future engineering reports are projections built from validated Insights.

Insights therefore represent the primary business product of DevLog AI.
