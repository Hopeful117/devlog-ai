# ADR-037 — Repository-First Context Extraction

- **Status:** Accepted
- **Date:** 2026-07-23

## Context

The quality of AI-generated analyses depends directly on the quality of the context provided to the model.

Software repositories already contain a significant amount of structured and semi-structured knowledge, including:

- source code;
- Git history;
- commit diffs;
- Architecture Decision Records;
- roadmap documents;
- README files;
- project documentation;
- tests;
- configuration files;
- build descriptors.

Many AI systems rely heavily on generic retrieval or external knowledge sources.

DevLog AI has a different objective.

Its primary responsibility is to understand the project itself before consulting any external context.

## Decision

DevLog AI adopts a **Repository-First** strategy.

The repository is considered the primary source of project knowledge.

Every analysis should maximize the exploitation of repository information before introducing additional external context.

External knowledge sources remain optional enrichments rather than primary evidence.

---

## Repository Context

The Repository Context represents every trustworthy information source available inside the project.

Examples include:

- source code;
- package structure;
- dependency graph;
- Git history;
- commit diff analyses;
- ADRs;
- roadmap items;
- validated Insights;
- deterministic analyses;
- AI task history;
- generated deliverables;
- README files;
- project conventions.

Each source contributes a different perspective on the project.

---

## Context Construction

Context construction should remain deterministic whenever possible.

The objective is not to send the entire repository to the LLM.

Instead, DevLog progressively builds a structured AnalysisContext composed of the most relevant repository evidence.

Selection criteria may include:

- repository location;
- semantic relevance;
- historical relationships;
- roadmap relationships;
- ADR references;
- dependency graph;
- recent changes;
- validated knowledge.

---

## Context Layers

Repository context should progressively be assembled in layers.

Typical layers include:

1. Current analysis target.
2. Related source code.
3. Git history.
4. Commit diff interpretations.
5. ADRs.
6. Roadmap items.
7. Validated Insights.
8. Previous analyses.
9. Project documentation.

Each layer enriches the context while remaining independently traceable.

---

## Deterministic Before AI

Whenever technically feasible, repository information should be extracted deterministically.

Examples include:

- dependency graphs;
- symbol extraction;
- REST endpoints;
- package relationships;
- configuration values;
- commit chronology;
- test coverage;
- architectural dependencies.

AI should primarily interpret structured repository knowledge rather than raw repository files.

---

## Context Budget

Repository context should remain bounded.

Selection mechanisms may include:

- ranking;
- chunking;
- summarization;
- evidence prioritization;
- token budgeting.

The objective is to maximize useful information rather than maximize token usage.

---

## Future Context Sources

Additional context sources may later be introduced.

Examples include:

- issue trackers;
- pull requests;
- CI/CD pipelines;
- monitoring systems;
- architecture diagrams;
- external documentation;
- chat discussions.

These sources must remain secondary.

Repository-derived knowledge always has priority when describing the project itself.

---

## Explainability

Every AI analysis should remain explainable.

DevLog should be capable of identifying which repository evidence contributed to each conclusion.

Repository evidence therefore remains traceable throughout the analysis lifecycle.

---

## Consequences

### Advantages

- higher quality analyses;
- reduced hallucinations;
- improved explainability;
- deterministic evidence selection;
- better project understanding;
- reusable analysis contexts.

### Trade-offs

- additional preprocessing;
- repository indexing cost;
- context ranking complexity.

These trade-offs are accepted because repository understanding is a fundamental capability of DevLog AI.

---

## Alternatives Considered

### External Knowledge First

Rejected.

External sources should enrich repository knowledge rather than replace it.

---

### Send Entire Repository to the LLM

Rejected.

Large repositories exceed model limits and reduce analysis quality.

Structured evidence selection provides better results.
