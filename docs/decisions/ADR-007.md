# ADR-007 - Analysis as the Analytical Aggregate

## Status

Accepted

## Date

2026-07-21

## Context

DevLog AI needs to persist the results of deterministic project analysis in a way that preserves context and traceability.

The analysis pipeline produces objective information at different levels:

* repository evidence,
* deterministic facts,
* deterministic observations.

These elements must remain associated with the analysis that produced them.

The system also needs to preserve the historical evolution of a project. A new analysis should therefore not overwrite the results of a previous analysis.

Introducing a separate `AnalysisRun` abstraction was considered, but the current V1 does not require a distinction between an analysis definition and an execution instance. Adding such an abstraction prematurely would increase architectural complexity without providing immediate value.

## Decision

DevLog AI will model `Analysis` as a persisted analytical aggregate containing the facts and observations produced during a specific analysis of a project.

The structure is:

```text
Project
   │
   └── Analysis
          │
          ├── Facts
          │
          └── Observations
```

Each `Analysis` represents a distinct analytical snapshot of the project.

A new analysis creates a new `Analysis` entity rather than modifying the results of a previous analysis.

For example:

```text
Project
   │
   ├── Analysis #1
   │      ├── Facts
   │      └── Observations
   │
   ├── Analysis #2
   │      ├── Facts
   │      └── Observations
   │
   └── Analysis #3
          ├── Facts
          └── Observations
```

`Facts` and `Observations` are therefore contextually associated with the `Analysis` that produced them.

The `Analysis` aggregate forms the deterministic analytical boundary between repository evidence and AI interpretation.

The resulting pipeline is:

```text
Repository Evidence
        │
        ▼
     Analysis
        │
   ┌────┴────┐
   ▼         ▼
 Facts   Observations
   │         │
   └────┬────┘
        ▼
  Analysis Context
        │
        ▼
    AI Engine
```

An analysis should be considered immutable once its processing is completed. Corrections or new information should result in a new analysis rather than arbitrary modification of historical analytical results.

## Alternatives Considered

### Separate Analysis and AnalysisRun entities

Introduce an `Analysis` entity representing an analysis definition and an `AnalysisRun` entity representing each execution.

Rejected for V1 because:

* the distinction is not currently required by the domain,
* it introduces additional complexity,
* it may prematurely model concepts that have not yet demonstrated a concrete need.

This approach may be reconsidered if DevLog AI later supports reusable analysis definitions, scheduled executions, or multiple execution strategies for the same analysis.

### Facts and Observations directly attached to Project

Store all facts and observations directly under the project without associating them with a specific analysis.

Rejected because:

* analytical provenance would be lost,
* it would be difficult to determine when and why information was detected,
* historical analytical states would be harder to reconstruct,
* comparison between analyses would become more difficult.

### Overwrite previous analysis results

Update existing facts and observations when a new analysis is performed.

Rejected because:

* historical information would be lost,
* changes over time could not be reliably reconstructed,
* AI analysis would lose access to previous project states,
* traceability would be reduced.

## Consequences

### Positive

* Preserves analytical history.
* Maintains clear provenance for facts and observations.
* Allows comparison between different project analyses.
* Provides a clear boundary between deterministic analysis and AI interpretation.
* Keeps the V1 domain model relatively simple.
* Enables future evolution toward more advanced analysis execution models.

### Negative

* Multiple analyses may contain overlapping or repeated facts.
* Historical data volume will increase over time.
* Comparing analyses requires explicit comparison logic.
* The model does not yet distinguish between an analysis definition and its execution.
