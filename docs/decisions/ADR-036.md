# ADR-036 — Commit-Level Code Diff Analysis

* **Status:** Accepted
* **Date:** 2026-07-23

## Context

ADR-035 introduces project-history understanding based on Git chronology and relationships with ADRs, roadmap items and trusted knowledge.

However, Git metadata and commit messages alone cannot reliably explain the technical evolution of a project.

Commit messages may be:

* incomplete;
* vague;
* incorrectly categorized;
* inconsistent with the actual change;
* focused on only one aspect of a mixed commit.

A commit named `fix stuff` may introduce an important architectural change, while a commit named `docs` may also modify runtime code.

DevLog must therefore analyse the actual code differences between commits.

## Decision

DevLog AI will introduce commit-level code-diff analysis.

The system will analyse the differences between a commit and its parent in order to identify the actual technical, functional and architectural changes introduced by that commit.

The diff and deterministically extracted code structure are considered more authoritative than the commit message.

The commit message remains contextual evidence describing the author's declared intent.

## Source Priority

When reconstructing a commit's meaning, DevLog follows this priority:

```text
Actual code diff
        >
Deterministically extracted code structure
        >
Trusted project knowledge
        >
Commit metadata and message
```

A commit message may guide interpretation but must never override contradictory code evidence.

## Analysis Pipeline

The target processing pipeline is:

```text
Commit and parent
        ↓
Git diff extraction
        ↓
Deterministic structural analysis
        ↓
AnalysisContext construction
        ↓
AI interpretation proposal
        ↓
Human or business validation
        ↓
Trusted historical knowledge
```

## Deterministic Extraction

Before invoking AI, DevLog should extract factual information whenever technically possible.

Examples include:

* files added, modified, renamed or removed;
* inserted and deleted lines;
* Java classes and interfaces added or removed;
* methods and fields added, modified or removed;
* annotations introduced or removed;
* REST endpoints changed;
* dependencies changed;
* database models changed;
* configuration properties changed;
* tests added or modified;
* frontend components, routes and services changed;
* documentation and ADR files changed.

Deterministic results become part of the `AnalysisContext`.

## Analysis Context

A commit diff analysis context should contain information equivalent to:

```java
public record CommitDiffAnalysisContext(
        String repositoryId,
        String commitHash,
        String parentHash,
        String commitMessage,
        Instant committedAt,
        List<ChangedFileContext> changedFiles,
        List<ChangedSymbolContext> changedSymbols,
        List<DependencyChangeContext> dependencyChanges,
        List<TestChangeContext> testChanges,
        DiffStatistics statistics,
        List<KnowledgeReference> relatedKnowledge,
        List<AdrReference> candidateAdrs,
        List<RoadmapReference> candidateRoadmapItems
) {
}
```

The exact implementation may evolve, but the context must remain structured, bounded and traceable.

## Interpretation Output

The AI Engine may propose a structured interpretation containing:

* technical summary;
* functional summary;
* change classification;
* affected components;
* introduced capabilities;
* modified behaviours;
* removed behaviours;
* architectural impact;
* possible breaking changes;
* testing impact;
* candidate ADR relationships;
* candidate roadmap relationships;
* confidence level;
* supporting evidence.

The output must follow a versioned and validated schema.

## Change Classification

Initial classifications may include:

* `FEATURE`;
* `FIX`;
* `REFACTORING`;
* `ARCHITECTURE`;
* `TEST`;
* `DOCUMENTATION`;
* `INFRASTRUCTURE`;
* `SECURITY`;
* `PERFORMANCE`;
* `MIXED`.

Classification is descriptive and should not be inferred solely from Conventional Commit prefixes.

## Message and Diff Divergence

DevLog should detect meaningful divergence between the declared commit message and the actual code changes.

Examples include:

* message describes documentation but production code changes;
* message describes a fix but introduces a new capability;
* message describes a refactoring but changes external behaviour;
* message omits a breaking contract change.

Such divergence should be represented as an observation or proposal, not silently ignored.

## Language-Aware Analysis

The first implementation may target the languages and frameworks used by DevLog AI's reference projects, including:

* Java and Spring Boot;
* TypeScript and Angular;
* Python;
* Markdown;
* YAML and configuration files.

The architecture must remain extensible to additional languages.

Language-specific analysers should produce a provider-independent structural representation.

## Diff Size Management

Large commits must not be blindly inserted into an AI prompt.

The system should support:

* file-level filtering;
* deterministic summarization;
* symbol-level extraction;
* chunking;
* relevance ranking;
* token-budget enforcement.

The original diff remains evidence but the AI receives a controlled `AnalysisContext`.

## Merge Commits

Merge commits require explicit handling.

Possible strategies include:

* analysing the first-parent diff;
* analysing differences against each parent;
* marking the commit as a merge event without duplicating already analysed changes.

The chosen strategy must remain traceable.

## Proposal Lifecycle

In accordance with ADR-006:

* AI interpretations are immutable proposals;
* proposals cannot modify project history directly;
* accepted interpretations are promoted atomically;
* rejected interpretations remain traceable;
* re-analysis creates a new proposal.

## Relationship With ADR-035

ADR-035 defines the historical model and chronology.

ADR-036 provides the technical evidence required to understand each commit.

Together they produce:

```text
Chronology and project context
              +
Actual code changes
              ↓
Trusted project history
```

## Expected Capabilities

DevLog should progressively become capable of:

* explaining what a commit actually changed;
* reconstructing history despite poor commit messages;
* detecting implementation of architectural decisions;
* identifying breaking changes;
* comparing declared intent with actual changes;
* grouping related commits into significant project events;
* explaining how a class, service or capability evolved.

## Consequences

### Advantages

* history based on actual code changes;
* reduced dependence on commit-message quality;
* more accurate project reports;
* stronger traceability;
* improved ADR and roadmap relationship detection;
* higher-quality article and documentation generation.

### Trade-offs

* language-specific parsing complexity;
* increased analysis cost;
* large-diff management;
* need to handle renames, generated files and merge commits;
* risk of incorrect AI interpretation requiring validation.

These trade-offs are accepted because code-diff analysis is necessary for trustworthy historical understanding.

## Alternatives Considered

### Analyse Commit Messages Only

Rejected.

Commit messages are useful but insufficient evidence.

### Send Raw Diffs Directly to the LLM

Rejected.

Raw diffs may be too large, noisy and vulnerable to embedded instructions. Deterministic preprocessing and structured context are required.

### Generate Trusted History Without Validation

Rejected.

AI interpretations must respect the existing proposal-validation architecture.
