# ADR-030 — Intent Execution and User Guidance

## Status

Accepted

## Date

2026-07-22

---

# Context

Previous ADRs established the deterministic knowledge pipeline and the Insight domain.

```text
Repository
    ↓
Workspace
    ↓
Collectors
    ↓
Facts
    ↓
Observations
    ↓
Project Profile
    ↓
AnalysisContext
    ↓
Intent
    ↓
InsightProposal
    ↓
Human Validation
    ↓
Insight
```

At this stage, DevLog AI possesses all deterministic knowledge required to generate high-quality Insight Proposals.

However, users rarely expect the same result for every request.

For example:

- describe the project;
- focus on architecture;
- write for recruiters;
- produce a concise explanation;
- generate a portfolio presentation.

These requests should influence the generated Insights without changing the underlying business contract.

DevLog AI therefore introduces the concept of **User Guidance**.

---

# Decision

Intent remains the unique business contract defining why the AI is invoked.

User Guidance allows users to refine the execution of an Intent without modifying its semantics.

The AI Engine therefore executes:

```text
Intent
+
User Guidance
+
AnalysisContext
        ↓
Prompt Builder
        ↓
Provider
        ↓
InsightProposal[]
```

---

# Intent

Intent defines the business objective.

Examples:

```text
describe-project-v1

generate-readme-v1

architecture-overview-v1
```

Intent determines:

- the expected Insight categories;
- the output schema;
- validation rules;
- business constraints;
- mandatory context.

Intent is deterministic and versioned.

---

# User Guidance

User Guidance represents optional user preferences.

It refines how an Intent should be executed.

It never changes:

- the business objective;
- allowed Insight categories;
- validation rules;
- provider output schema;
- deterministic constraints.

---

# Responsibilities

User Guidance may influence:

## Focus

Example:

> Focus on the distributed architecture.

---

## Audience

Examples:

> Explain for junior developers.

> Explain for recruiters.

---

## Level of Detail

Examples:

> Keep it concise.

> Produce a detailed explanation.

---

## Writing Style

Examples:

> Use a technical style.

> Use a pedagogical tone.

---

## Output Context

Examples:

> This will be published on my portfolio.

> This will become internal documentation.

---

## Priorities

Examples:

> Explain Docker before Spring Boot.

---

# Non-Responsibilities

User Guidance cannot:

- request unsupported Insight categories;
- bypass deterministic knowledge;
- ignore Facts or Observations;
- invent project characteristics;
- modify the output schema;
- disable validation rules;
- request arbitrary prompt execution.

---

# Priority Order

Prompt construction follows a strict precedence.

```text
Intent
        ↓
AnalysisContext
        ↓
User Guidance
```

Intent always has the highest priority.

AnalysisContext provides the trusted project knowledge.

User Guidance only influences presentation and emphasis.

---

# Prompt Builder

Prompt Builder combines:

```text
Intent

+

AnalysisContext

+

User Guidance
```

to build the provider-specific prompt.

Prompt Builder remains an implementation component.

Business semantics belong exclusively to Intent and User Guidance.

---

# Deterministic Constraints

Regardless of User Guidance, the AI must remain grounded on the AnalysisContext.

Every generated InsightProposal must be supported by deterministic project knowledge.

User Guidance must never encourage hallucination.

---

# Provider Independence

User Guidance is provider-agnostic.

OpenAI, Anthropic or future providers must receive equivalent semantic information.

Prompt wording remains provider-specific.

User Guidance remains provider-independent.

---

# Human Validation

User Guidance has no impact on validation.

Generated InsightProposals remain:

- proposed;
- reviewable;
- rejectable;
- traceable.

Only accepted proposals become Insights.

---

# Prompt Request

Conceptually:

```text
PromptRequest
├── Intent
├── User Guidance
└── AnalysisContext
```

PromptRequest represents the complete input of the Prompt Builder.

It is not exposed as a business object outside the AI Engine boundary.

---

# Examples

Example:

```text
Intent

describe-project-v1
```

User Guidance:

```text
Explain this project for my portfolio.
Focus on the distributed architecture.
Keep the explanation under 800 words.
```

Expected result:

- same Insight categories;
- same deterministic constraints;
- different emphasis and writing style.

---

# Architectural Principle

Intent defines **why** the AI is invoked.

AnalysisContext defines **what** the AI knows.

User Guidance defines **how** the user wishes to consume the result.

Prompt Builder transforms these three elements into a provider-specific prompt.

The provider never determines the business objective.

---

# Consequences

## Positive

- Clear separation between business contract and user preferences.
- Strong provider independence.
- Stable prompt architecture.
- Better prompt reuse.
- Reduced prompt duplication.
- Easier testing.
- Better future UI integration.

## Negative

- Additional abstraction.
- Requires validation of supported guidance fields.
- Prompt Builder becomes slightly more sophisticated.

---

# Alternatives Considered

## Free-form user prompts

Rejected.

Business semantics become coupled to prompt wording.

---

## Intent only

Rejected.

Too rigid for real-world documentation workflows.

---

## User Guidance overriding Intent

Rejected.

Business contracts must remain deterministic.

---

# Fundamental Rule

> **User Guidance refines the execution of an Intent but never alters its business contract.**
