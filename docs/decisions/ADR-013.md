# ADR-013 - Deterministic Construction of Analysis Context

## Status

Accepted

## Date

2026-07-21

## Context

DevLog AI uses the Java Core as the authoritative owner of project knowledge and deterministic analysis.

The Python AI Engine is responsible for interpretation and contextual reasoning, but it should not independently determine which project knowledge is authoritative or relevant to a business workflow.

AI tasks require structured context containing the information necessary for the requested interpretation.

Providing the entire project knowledge base to the AI Engine would introduce unnecessary:

* token usage,
* processing cost,
* latency,
* noise,
* ambiguity.

Conversely, allowing the AI Engine to independently retrieve and select project knowledge would move part of the knowledge retrieval responsibility outside the Core and increase coupling between AI processing and the domain model.

The system therefore needs a deterministic mechanism for constructing a focused context before AI processing begins.

## Decision

The Java Core will construct `AnalysisContext` deterministically before submitting an AI task.

The AI Engine will receive the prepared context and will not independently retrieve project knowledge from the Core's persistence layer.

The conceptual workflow is:

```text id="j3q8m1"
AI Task
   │
   ▼
Context Retrieval
   │
   ▼
Context Builder
   │
   ├── Facts
   ├── Observations
   ├── Validated Knowledge
   ├── Related Analyses
   └── Relevant Evidence
   │
   ▼
AnalysisContext
   │
   ▼
AI Engine
```

The `AnalysisContext` is a boundary object between the deterministic Core and the AI Engine.

It contains only the information considered relevant for the specific AI task.

## Context Selection

The Core will select context according to the nature of the requested AI task.

For example, an engineering decision analysis may require:

```text id="r7m4p2"
Current Analysis
Relevant Facts
Relevant Observations
Related Engineering Decisions
Related Evidence
Recent Project Changes
```

A project summary may require:

```text id="k2v8x5"
Recent Analyses
Milestones
Validated Events
Validated Decisions
Challenges
Project State
```

A social media post generation task may require:

```text id="m6q3z9"
Recent Milestones
Validated Events
Selected Achievements
Project Description
Technology Stack
```

The context therefore depends on the task being executed.

## Context Builder

The Core will provide a dedicated context construction mechanism responsible for assembling the context required by an AI task.

Conceptually:

```text id="n5j2c7"
AI Task
    │
    ▼
Context Builder
    │
    ├── Retrieve relevant data
    ├── Apply deterministic filters
    ├── Resolve relationships
    ├── Select supporting evidence
    └── Assemble structured context
    │
    ▼
AnalysisContext
```

The exact implementation may use task-specific builders, profiles, or strategies as the system evolves.

However, the responsibility remains within the Core.

## Structured Context Principle

The context sent to the AI Engine should be structured rather than an unbounded collection of raw project data.

Conceptually:

```text id="v9k4s1"
AnalysisContext
├── Task Information
├── Project Context
├── Current Analysis
├── Facts
├── Observations
├── Relevant Knowledge
├── Evidence
└── Constraints
```

The exact schema will evolve according to the types of AI tasks supported by DevLog AI.

The Core should provide sufficient information for the AI Engine to perform its task without requiring direct access to the underlying project database.

## Alternatives Considered

### Send the entire project knowledge base

Provide all available project knowledge to the AI Engine for every task.

Rejected because:

* context size would grow continuously,
* token usage and processing costs would increase,
* irrelevant information would introduce noise,
* the AI Engine would need to determine relevance itself,
* context quality would become harder to control.

### Allow the AI Engine to retrieve knowledge directly

Expose Core APIs or persistence access that allow Python to retrieve additional knowledge during processing.

Rejected because:

* the AI Engine would become coupled to the Core domain model,
* knowledge retrieval responsibility would move outside the Core,
* AI processing would become dependent on Core persistence structures,
* the boundary between domain ownership and AI processing would become less clear.

### Let the AI Engine determine context relevance

Provide a large context and allow Python to select the relevant information.

Rejected because:

* relevance selection would become an AI responsibility,
* unnecessary information would still need to be transmitted,
* deterministic filtering could be performed more reliably by the Core,
* the AI Engine would gain unnecessary responsibility over project knowledge.

## Consequences

### Positive

* Keeps project knowledge ownership within the Core.
* Reduces unnecessary context sent to the AI Engine.
* Improves AI processing quality by reducing irrelevant information.
* Reduces token usage and processing cost.
* Makes AI tasks more deterministic and reproducible.
* Allows the AI Engine to evolve independently from Core persistence structures.
* Provides a clear contract between Core and AI Engine.

### Negative

* Context construction adds complexity to the Core.
* The Core must implement task-specific retrieval and filtering logic.
* Context quality depends on the quality of deterministic retrieval.
* Future context selection improvements may require additional retrieval mechanisms.
