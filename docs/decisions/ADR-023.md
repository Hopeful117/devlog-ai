# ADR-023 — Deterministic Knowledge Collection Pipeline

## Status

Accepted

## Date

2026-07-22

---

# Context

DevLog AI aims to build a long-term engineering memory from software projects.

Previous ADRs established the following principles:

- the Java Core owns project knowledge;
- deterministic processing always precedes AI interpretation;
- the AI Engine only interprets an immutable `AnalysisContext`;
- trusted knowledge is always validated by the Core.

The next architectural step is to define how deterministic knowledge is extracted from a project before any AI interaction.

The objective is to establish a scalable collection pipeline capable of supporting future knowledge sources while remaining deterministic and reproducible.

---

# Decision

DevLog AI introduces a **Deterministic Knowledge Collection Pipeline**.

Knowledge acquisition is decomposed into independent stages:

```text
Workspace
        │
        ▼
Collectors
        │
        ▼
Facts
        │
        ▼
Observation Engine
        │
        ▼
Observations
        │
        ▼
AnalysisContext
        │
        ▼
AI Engine
```

Each stage owns a single responsibility.

---

# Initial Knowledge Source

Version 1 supports a single knowledge source:

- Git repositories.

Additional sources (GitHub, GitLab, Jira, Confluence, CI/CD, etc.) are intentionally excluded from this ADR and may be introduced later without changing the pipeline architecture.

---

# Persistent Workspace

Each `Source` owns a persistent local workspace.

The workspace is:

- created during the first analysis;
- reused for subsequent analyses;
- synchronized before every analysis;
- isolated from the business domain.

The workspace is considered an infrastructure concern and never part of the project knowledge.

---

# Workspace Synchronization

Before every analysis, the Core synchronizes the workspace deterministically.

The synchronization sequence is:

```text
Fetch
    ↓
Clean
    ↓
Checkout
    ↓
Reset --hard
    ↓
Analysis
```

If the workspace is missing or corrupted, the Core recreates it automatically.

Only one synchronization may modify a given workspace simultaneously.

---

# Analysis Revision

An analysis targets a Git revision.

By default:

```text
defaultBranch@HEAD
```

An explicit revision may also be provided:

- branch;
- tag;
- commit SHA.

This allows deterministic reproduction of previous analyses.

---

# Collectors

Knowledge extraction is performed by independent **Collectors**.

Examples:

- GitCollector
- BuildCollector
- SpringCollector
- DockerCollector
- DocumentationCollector
- TestCollector

Each collector has a single responsibility.

Collectors never communicate directly with each other.

---

# Collector Responsibility

Collectors produce only deterministic **Facts**.

They never create:

- Observations;
- Insights;
- Recommendations;
- AI proposals;
- Business knowledge.

Example:

```text
Workspace

↓

GitCollector

↓

Fact
```

---

# Collector Principles

Every collector must satisfy the following properties.

## Deterministic

The same workspace always produces the same facts.

```text
Same Workspace
        ↓
Same Collector
        ↓
Same Facts
```

---

## Independent

Collectors operate independently.

Adding or removing a collector must not require modifications to existing collectors.

---

## Idempotent

Running the same collector multiple times on an unchanged workspace produces identical results.

---

## Stateless

Collectors do not maintain project state.

Persistent knowledge belongs exclusively to the Java Core.

---

# Observation Engine

The Observation Engine consumes deterministic Facts.

Its responsibility is to derive higher-level deterministic Observations.

```text
Facts
        │
        ▼
Observation Engine
        │
        ▼
Observations
```

The Observation Engine never accesses Git repositories directly.

It operates exclusively on Facts.

---

# Analysis Context

The AnalysisContext is constructed from:

- Facts;
- Observations;
- project metadata;
- validated project knowledge.

The AI Engine never accesses:

- Git repositories;
- workspaces;
- databases;
- project files.

Its only input remains:

```text
AnalysisContext
```

---

# AI Boundary

The AI Engine remains responsible only for interpretation.

It never:

- clones repositories;
- parses source code;
- analyses Maven files;
- analyses Dockerfiles;
- discovers project information by itself.

Its responsibility starts only after deterministic context construction.

---

# Architectural Principles

The pipeline follows the rule:

```text
Collect
        ↓
Observe
        ↓
Interpret
        ↓
Validate
        ↓
Knowledge
```

Deterministic processing always precedes AI interpretation.

The AI never replaces deterministic analysis.

---

# Consequences

## Positive

- Strong separation of responsibilities.
- Deterministic and reproducible analyses.
- Extensible collector architecture.
- Provider-independent knowledge acquisition.
- Better scalability for future project sources.
- Richer AnalysisContext for AI interpretation.
- Easier testing of deterministic components.

## Negative

- Additional architectural components.
- Slightly more complex orchestration.
- More infrastructure to maintain (workspace lifecycle).

---

# Alternatives Considered

## Monolithic Git Scanner

Rejected.

A single scanner would accumulate multiple responsibilities and become difficult to evolve.

---

## AI-driven Repository Analysis

Rejected.

AI must never replace deterministic knowledge collection.

---

## Temporary Workspace

Rejected.

Re-cloning repositories for every analysis would significantly reduce performance on large projects.

Persistent workspaces provide better scalability while preserving deterministic synchronization.

---

# Future Evolution

Future versions may introduce additional collectors for:

- GitHub;
- GitLab;
- Jira;
- Confluence;
- CI/CD platforms;
- Kubernetes;
- Terraform;
- Cloud infrastructure.

These additions will enrich the collection pipeline without modifying its overall architecture.
