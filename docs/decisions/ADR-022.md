# ADR-022 - Initial LLM Integration and Structured Interpretation Pipeline

## Status

Accepted

## Date

2026-07-21

## Context

The distributed architecture introduced by ADR-003 and completed through ADR-020 establishes a clear separation between the Java Core and the Python AI Engine.

The Java Core is responsible for:

* project knowledge;
* deterministic analysis;
* workflow orchestration;
* task lifecycle;
* validation;
* persistence.

The AI Engine is responsible for interpreting an immutable `AnalysisContext` and producing structured AI proposals.

The first real LLM integration must preserve these architectural principles while remaining provider-independent and fully testable.

## Decision

DevLog AI introduces its first LLM capability through a provider abstraction dedicated to structured interpretation.

The first supported capability is:

```text
INSIGHT_GENERATION
```

The AI Engine remains an interpretation service.

It never becomes responsible for project knowledge or business decisions.

## Architectural Principle

The Java Core owns:

* knowledge;
* context construction;
* orchestration;
* validation;
* persistence.

The Python AI Engine owns:

* prompt construction;
* LLM invocation;
* structured interpretation.

The LLM owns:

* language understanding only.

The LLM never becomes part of the domain model.

It is an implementation detail hidden behind the `LlmProvider` abstraction.

## Context Isolation

The AI Engine is intentionally context-blind.

It never accesses:

* PostgreSQL;
* Git repositories;
* Git providers;
* project files;
* project APIs.

Its only input is:

```text
AnalysisContext
```

produced by the Java Core.

The AI Engine must never collect project information by itself.

```text
Analysis

```

## First Supported Task

The first supported AI task is:

```text
INSIGHT_GENERATION
```

Its objective is to generate engineering insights derived from the deterministic analysis context.

It must not generate:

* architecture decisions;
* trusted knowledge;
* project modifications.

## Provider Abstraction

The AI Engine communicates only through:

```python
class LlmProvider(Protocol):

    async def generate_structured(
        self,
        request: LlmRequest,
        response_model: type[T],
    ) -> T:
        ...
```

Business services must never depend directly on an OpenAI SDK.

Future providers may include:

* OpenAI;
* Anthropic;
* Gemini;
* Local models.

Replacing the provider must not require changing business logic.

## Initial Provider

The first implementation uses:

```text
OpenAI
```

The provider implementation remains isolated behind the abstraction.

## Prompt Builder

Prompt generation is the responsibility of a dedicated component.

```text
AnalysisContext
        │
        ▼
InsightPromptBuilder
        │
        ▼
Prompt
        │
        ▼
LlmProvider
```

Providers never build prompts.

Prompt construction is deterministic and versioned.

The initial prompt version is:

```text
insight-generation-v1
```

## Prompt Structure

The prompt consists of four sections:

* System Instructions
* Task Definition
* Structured Analysis Context
* Expected Structured Output

The prompt explicitly instructs the model to reason only from the provided context.

If evidence is insufficient, the model must express uncertainty rather than invent information.

## Structured Output

The LLM must return structured data validated using Pydantic.

Example:

```text
InsightGenerationOutput

└── proposals[]

        ├── title
        ├── summary
        ├── rationale
        ├── confidence
        ├── supportingFactIds
        ├── supportingObservationIds
        └── evidenceReferences
```

The AI Engine must never parse arbitrary free-form responses.

## Validation

Validation occurs in two stages.

### Python

The AI Engine validates:

* schema;
* required fields;
* confidence range;
* referenced identifiers;
* payload integrity.

### Java Core

The Core validates:

* task existence;
* analysis consistency;
* ownership;
* proposal type;
* business rules.

The callback is never blindly trusted.

## Output Ownership

The callback creates only:

```text
ValidatableProposal
```

It never creates:

* Insight;
* Decision;
* KnowledgeEvent;
* Documentation;
* trusted project knowledge.

## Failure Strategy

If the LLM produces invalid structured output:

1. validate with Pydantic;
2. perform one corrective retry;
3. if validation still fails:

```text
AiTask → FAILED
```

Only one corrective retry is permitted.

Technical retries are handled separately.

## Configuration

The provider configuration is externalized.

Initial settings:

```text
LLM_PROVIDER
LLM_MODEL
LLM_API_KEY
LLM_TIMEOUT_SECONDS
LLM_MAX_OUTPUT_TOKENS
```

The default development configuration is:

```text
LLM_PROVIDER=mock
```

The AI Engine must remain executable without external API credentials.

## Testing

Tests must cover:

* mock provider;
* successful structured generation;
* invalid output;
* corrective retry;
* failed retry;
* callback generation;
* provider substitution.

No automated test requires access to the real OpenAI API.

## Alternatives Considered

### Direct SDK usage inside services

Rejected.

Business logic must not depend on a provider SDK.

### LLM builds project knowledge directly

Rejected.

Only the Java Core owns trusted knowledge.

### AI Engine accesses repositories

Rejected.

Repository access belongs exclusively to the Java Core.

### Free-form text output

Rejected.

Structured outputs provide deterministic validation and stable contracts.

## Consequences

### Positive

* Strong separation of concerns.
* Provider independence.
* Replaceable LLM implementations.
* Deterministic validation.
* Testability without external services.
* Stable contracts between Java and Python.
* Preservation of Core ownership.

### Negative

* Additional abstraction layer.
* Prompt engineering becomes an explicit responsibility.
* Structured outputs require schema maintenance.
* Two-stage validation increases implementation effort.

## Architectural Principle

The architecture follows the rule:

```text
Java Core
    │
    ▼
Owns knowledge

Python AI Engine
    │
    ▼
Owns interpretation

LLM
    │
    ▼
Owns language generation only
```

The AI Engine is intentionally context-blind.

Its only input is an immutable `AnalysisContext`.

Its only output is a structured proposal.
