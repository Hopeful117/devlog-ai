# ADR-026 — Deterministic Observation Model

## Status

Accepted

## Date

2026-07-22

---

# Context

Previous ADRs established the deterministic knowledge acquisition pipeline.

```text
Repository
    ↓
Workspace
    ↓
Collectors
    ↓
Facts
    ↓
Observation Engine
    ↓
AnalysisContext
    ↓
AI Engine
    ↓
ValidatableProposal
```

Collectors are now capable of extracting deterministic Facts from real software projects.

However, the role of the Observation Engine has not yet been formally defined.

This ADR establishes the semantic boundary between deterministic knowledge acquisition and AI interpretation.

---

# Decision

DevLog AI introduces a deterministic Observation layer.

The Observation Engine is responsible for transforming low-level technical Facts into higher-level objective project characteristics.

Observations become the semantic language shared between deterministic analysis, AI interpretation and human validation.

---

# Knowledge Hierarchy

DevLog AI represents project knowledge through successive abstraction levels.

```text
Evidence
        │
        ▼
Facts
        │
        ▼
Observations
        │
        ▼
Insights
        │
        ▼
Validated Knowledge
```

Each layer increases abstraction while preserving traceability.

---

# Evidence

Evidence represents raw technical artifacts.

Examples:

```text
Dockerfile
pom.xml
README.md
@RestController
application.yml
```

Evidence is never interpreted directly by AI.

---

# Facts

Collectors transform Evidence into deterministic Facts.

Examples:

```text
SPRING_BOOT_PRESENT

DOCKERFILE_PRESENT

README_PRESENT

REST_CONTROLLER_DECLARED
```

Facts remain objective.

They never describe project meaning.

---

# Observation Definition

An Observation is:

> **A deterministic, reproducible and objectively demonstrable project characteristic derived from one or more traceable Facts through an explicitly defined rule.**

An Observation always satisfies the following properties.

---

# Deterministic

The same Facts always produce the same Observation.

```text
Same Facts

↓

Same Observation
```

---

# Reproducible

Observation generation is entirely deterministic.

No probabilistic reasoning or AI interpretation participates.

---

# Traceable

Every Observation must explain why it exists.

Conceptually:

```text
Observation

↓

Observation Rule

↓

Facts

↓

Evidence
```

There is never an unexplained Observation.

---

# Objective

Observations describe observable project characteristics.

They never infer:

- developer intentions;
- software quality;
- architectural maturity;
- future evolution.

Examples.

Valid:

```text
The project exposes REST endpoints.
```

Invalid:

```text
The project has a good REST architecture.
```

---

# Higher Level Than Facts

Observations increase abstraction.

Example.

Facts:

```text
@RestController detected

@RequestMapping detected
```

Observation:

```text
The project exposes an HTTP REST API.
```

---

# Observation Rules

Observations are produced exclusively through deterministic Observation Rules.

Conceptually:

```text
Observation Rule

Conditions

↓

Facts

↓

Evaluation

↓

Observation
```

Rules remain explicit, testable and versioned.

---

# Observation Independence

Observations must not depend directly on other Observations.

Preferred:

```text
Facts

↓

Observation A

Facts

↓

Observation B
```

Avoid:

```text
Observation A

↓

Observation B

↓

Observation C
```

The Observation Engine should remain a shallow deterministic reasoning layer.

Complex interpretation belongs to the AI Engine.

---

# Temporal Scope

Observations describe the analysed project at the analysed revision.

They never describe:

- future evolution;
- recommendations;
- expected changes;
- possible improvements.

They always describe:

```text
Current repository state
```

---

# No Quality Judgement

Observations never express opinions.

Forbidden examples:

```text
Good architecture

Poor documentation

Bad practices
```

Those conclusions belong to AI-generated Insights or human review.

---

# Human Value

An Observation must provide information beyond a single isolated Fact.

Simply renaming a Fact does not create an Observation.

Observation exists to reduce technical complexity into understandable project characteristics.

---

# Semantic Layer

The Observation Engine forms the deterministic semantic layer of DevLog AI.

Collectors describe:

```text
What exists.
```

Observation Engine describes:

```text
What can be objectively concluded.
```

AI describes:

```text
What it may mean.
```

Humans decide:

```text
What becomes trusted knowledge.
```

Conceptually:

```text
Facts

↓

What exists?

↓

Observations

↓

What can be objectively stated?

↓

Insights

↓

What does it probably mean?

↓

Human Validation

↓

Trusted Knowledge
```

---

# AnalysisContext

The AI Engine never consumes raw repository structure directly.

Instead, it receives:

- Facts;
- Observations;
- validated project knowledge;
- project metadata.

Observations therefore become the primary semantic language of AnalysisContext.

---

# Examples

Example 1

Facts:

```text
Dockerfile present

docker-compose.yml present
```

Observation:

```text
Project is containerized.
```

---

Example 2

Facts:

```text
Spring Boot dependency

@RestController

SpringApplication
```

Observation:

```text
Project exposes a Spring Boot REST application.
```

---

Example 3

Facts:

```text
ADR directory present

Architecture documentation detected
```

Observation:

```text
Project contains architecture documentation.
```

---

# Non Examples

Invalid Observation:

```text
The project is production ready.
```

Reason:

Requires subjective interpretation.

---

Invalid Observation:

```text
Developers chose microservices because...
```

Reason:

Developer intention cannot be determined deterministically.

---

Invalid Observation:

```text
The project should migrate to Java 25.
```

Reason:

Recommendation belongs to AI.

---

# Design Principles

Observation Engine must remain:

- deterministic;
- explainable;
- reproducible;
- traceable;
- shallow;
- objective;
- implementation-independent.

---

# Responsibilities

Collectors:

```text
Extract Facts.
```

Observation Engine:

```text
Produce semantic project characteristics.
```

AI Engine:

```text
Interpret project characteristics.
```

Humans:

```text
Validate knowledge.
```

---

# Consequences

## Positive

- Clear semantic separation.
- Better explainability.
- Higher quality AI prompts.
- Easier deterministic testing.
- Stable project vocabulary.
- Reduced prompt complexity.
- Better long-term maintainability.

## Negative

- Additional reasoning layer.
- New Observation Rules to maintain.
- Careful control of abstraction level required.

---

# Architectural Principle

The Observation Engine is the deterministic semantic layer of DevLog AI.

It transforms low-level technical evidence into objective project characteristics that can be interpreted consistently by both humans and AI models.
