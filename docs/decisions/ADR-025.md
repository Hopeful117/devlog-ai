# ADR-025 — Analysis Diagnostics and Pipeline Observability

## Status

Accepted

## Date

2026-07-22

---

# Context

Previous ADRs established the complete deterministic analysis pipeline:

```text
Source
    ↓
Workspace
    ↓
Collectors
    ↓
Facts
    ↓
Observation Engine
    ↓
Observations
    ↓
AnalysisContext
    ↓
AI Engine
    ↓
ValidatableProposal
```

The pipeline is now capable of analysing real repositories.

However, there is currently no unified mechanism allowing developers or future users to understand:

- what happened during an Analysis;
- whether the collection was complete;
- which Collectors participated;
- which resources were generated;
- which data was actually transmitted to the AI Engine.

Debugging currently requires consulting logs or querying multiple repositories directly.

A dedicated diagnostics layer is required.

---

# Decision

DevLog AI introduces an immutable diagnostics model attached to every Analysis.

Diagnostics become the primary entry point for understanding an Analysis execution.

The diagnostics endpoint answers the question:

> **"How did this Analysis execute?"**

It does not replace dedicated resources such as Facts or Observations.

---

# Hybrid API

Diagnostics follow a hybrid API.

A lightweight aggregated endpoint provides the global execution summary.

```http
GET /api/v1/analyses/{analysisId}/diagnostics
```

Dedicated resources remain available through specialized endpoints.

Examples:

```http
GET /api/v1/analyses/{analysisId}/facts
GET /api/v1/analyses/{analysisId}/observations
GET /api/v1/analyses/{analysisId}/warnings
GET /api/v1/analyses/{analysisId}/context
GET /api/v1/analyses/{analysisId}/ai-task
GET /api/v1/analyses/{analysisId}/proposals
```

The diagnostics endpoint must never embed complete collections.

It provides a stable summary and references to detailed resources.

---

# Analysis Diagnostics Model

The diagnostics response is organized into stable sections.

```text
AnalysisDiagnostics
│
├── Identity
├── Revision
├── Timeline
├── Counts
├── Completeness
├── Collector Summary
├── AI Task Summary
├── Pipeline Summary
├── Technical Metadata
└── Resource Links
```

---

# Identity

Identity describes the analysed resource.

Example fields:

```text
analysisId
projectId
analysisType
status
```

---

# Revision

Revision identifies exactly what was analysed.

Example:

```text
requestedRevision
resolvedRevision
```

The diagnostics always expose the resolved Git revision actually analysed.

---

# Timeline

Timeline provides execution timing.

Example fields:

```text
createdAt
startedAt
completedAt
duration
```

The diagnostics expose total execution duration.

Fine-grained collector timings remain outside the scope of V1.

---

# Counts

Diagnostics expose aggregate resource counts.

Examples:

```text
sourceCount
factCount
observationCount
warningCount
proposalCount
```

The diagnostics never embed the resources themselves.

---

# Completeness

Diagnostics expose the quality of the deterministic collection.

Examples:

```text
collectionComplete
truncated
warningCount
errorCount
```

Collection completeness is intentionally separated from Analysis status.

An Analysis may complete successfully while the collected context remains partially incomplete.

---

# Collector Summary

Diagnostics summarize collector execution.

Examples:

```text
collectorCount
successfulCollectors
collectorsWithWarnings
failedCollectors
```

Detailed collector outputs remain accessible through dedicated resources.

---

# AI Task Summary

Diagnostics expose only the execution state of the AI workflow.

Examples:

```text
taskType
status
provider
startedAt
completedAt
```

Diagnostics never expose:

- prompts;
- raw provider responses;
- API keys;
- provider-specific internal metadata.

---

# Pipeline Summary

Diagnostics expose a concise execution overview.

Conceptually:

```text
Workspace
    ↓
Collectors
    ↓
Facts
    ↓
Observations
    ↓
AnalysisContext
    ↓
AI Task
    ↓
Proposals
```

The pipeline summary is informational.

It does not replace individual resources.

---

# Technical Metadata

Diagnostics expose stable technical metadata useful for reproducibility.

Examples:

```text
contextBuilderVersion
collectorVersions
serializedContextSize
```

This metadata supports future comparisons between analyses performed with different collector implementations.

---

# Resource Links

Diagnostics provide references to detailed resources.

Examples:

```text
facts
observations
warnings
context
aiTask
proposals
```

The exact representation is implementation-dependent.

The objective is to simplify navigation between related resources.

---

# AnalysisContext Exposure

The AnalysisContext becomes observable through a dedicated endpoint.

```http
GET /api/v1/analyses/{analysisId}/context
```

The endpoint returns:

- the immutable snapshot actually transmitted to the AI Engine;
- exactly the context used during interpretation.

The endpoint never rebuilds the context dynamically.

This guarantees reproducibility.

Future authorization rules may restrict access.

---

# Collection Warnings

Collector warnings become persistent Analysis resources.

Warnings describe the execution of deterministic collection.

They do **not** describe the analysed project.

Examples:

```text
FILE_LIMIT_REACHED

PARTIALLY_SUPPORTED_FORMAT

COLLECTOR_TIMEOUT

SYMBOLIC_LINK_SKIPPED
```

Warnings are therefore neither:

- Facts;
- Observations;
- Insights;
- validated knowledge.

They represent metadata describing the quality and completeness of the collection.

---

# Warning Model

Conceptually:

```text
CollectionWarning
│
├── analysisId
├── sourceId
├── collectorType
├── collectorVersion
├── code
├── severity
├── message
├── evidenceReference
├── metadata
└── occurredAt
```

Warning codes must remain stable.

Human-readable messages may evolve.

---

# Warning Usage

Warnings serve multiple purposes.

They support:

- diagnostics;
- collector improvement;
- comparison between analyses;
- evaluation of context completeness.

A bounded warning summary may become part of the AnalysisContext.

The AI Engine may therefore adapt its confidence without receiving raw diagnostic noise.

---

# AnalysisContext Principle

The AI Engine must always receive the same immutable snapshot exposed through diagnostics.

Consequently:

```text
Persist Snapshot

↓

Send Snapshot

↓

Expose Snapshot
```

The diagnostics endpoint never reconstructs AnalysisContext from current project knowledge.

---

# Design Principles

Diagnostics are:

- immutable;
- reproducible;
- lightweight;
- implementation-independent.

Diagnostics summarize execution.

Dedicated resources provide details.

---

# Exclusions

Diagnostics intentionally exclude:

- complete Fact lists;
- complete Observation lists;
- raw prompts;
- raw LLM responses;
- stack traces;
- application logs;
- collector internal state;
- repository contents.

Those concerns belong to specialized endpoints.

---

# Future UI

Diagnostics are designed to become the foundation of the future Analysis dashboard.

A typical UI may display:

```text
Analysis
│
├── Status
├── Revision
├── Duration
├── Pipeline Summary
├── Counts
├── Warnings
└── Resource Links
```

The diagnostics API therefore becomes the primary entry point for visualizing Analysis execution.

---

# Consequences

## Positive

- Single entry point for Analysis execution.
- Easier debugging.
- Improved reproducibility.
- Better AI prompt engineering.
- Clear separation between summaries and resources.
- Stable contract for future frontend development.
- Better visibility into deterministic collection quality.

## Negative

- Additional DTOs.
- Additional persistence for warnings.
- Larger API surface.
- Future versioning of diagnostics metadata.

---

# Alternatives Considered

## Logs only

Rejected.

Logs are implementation artifacts.

They cannot serve as a stable diagnostics API.

---

## Diagnostics containing every resource

Rejected.

Large payloads would reduce performance and blur resource boundaries.

---

## Dynamic AnalysisContext reconstruction

Rejected.

Only the persisted immutable snapshot guarantees reproducibility.

---

# Architectural Principle

Diagnostics explain execution.

Facts describe repositories.

Observations describe deterministic interpretations.

The AI interprets the resulting context.

Knowledge remains validated by humans.
