# ADR-018 - Analysis Workflow Orchestration

## Status

Accepted

## Date

2026-07-21

## Context

DevLog AI now contains the primary domain components required to prepare AI processing:

* `Analysis`;
* deterministic `Fact` objects;
* deterministic `Observation` objects;
* `AnalysisContext`;
* `AiTask`;
* `ValidatableProposal`;
* human `Validation`.

These components currently expose their own domain services, but no application-level component coordinates them into a complete analytical workflow.

Without an explicit orchestration layer, controllers or individual domain services could progressively assume cross-domain responsibilities such as:

* starting an analysis;
* generating deterministic results;
* building an AI context;
* creating an AI task;
* submitting work to the AI Engine;
* handling failures across several domains.

This would distribute workflow logic across controllers and domain services and make the complete analysis process difficult to understand, test, and evolve.

DevLog AI therefore needs a dedicated application service responsible for coordinating the analysis workflow while preserving the responsibility boundaries established by the existing ADRs.

## Decision

DevLog AI will introduce an `AnalysisWorkflowService` in the Java Core.

The service will orchestrate the application-level workflow that transforms an analysis request into an AI task prepared for submission.

The initial workflow is:

```text
Trigger
   │
   ▼
AnalysisWorkflowService
   │
   ├── Create or load Analysis
   ├── Start Analysis
   ├── Run deterministic analysis
   ├── Persist Facts
   ├── Persist Observations
   ├── Build AnalysisContext
   ├── Create AiTask
   └── Prepare future AI submission
```

The workflow service coordinates specialized services but does not replace them.

Conceptually:

```text
AnalysisWorkflowService
├── AnalysisService
├── DeterministicAnalysisService
├── AnalysisContextService
├── AiTaskService
└── AIEngineClient          future increment
```

## Responsibilities

### AnalysisWorkflowService

The workflow service is responsible for:

* coordinating the complete analytical use case;
* enforcing the ordering of workflow steps;
* validating whether an analysis may be started;
* updating the business status of the analysis;
* invoking deterministic analysis capabilities;
* requesting construction of `AnalysisContext`;
* creating the corresponding `AiTask`;
* coordinating failure handling between workflow steps;
* returning a workflow-level result.

It answers:

> What must happen, and in which order, to prepare an analysis for AI processing?

### Specialized services

Specialized services retain ownership of their own domain behavior.

#### AnalysisService

Responsible for:

* creating analyses;
* loading analyses;
* validating analysis state;
* applying analysis state transitions;
* persisting analysis lifecycle changes.

#### DeterministicAnalysisService

Responsible for:

* extracting objective Facts;
* deriving deterministic Observations;
* preserving evidence references;
* persisting deterministic analysis results.

#### AnalysisContextService

Responsible for:

* deterministic context retrieval;
* bounded and ordered data selection;
* building immutable context snapshots;
* applying context-selection policies.

#### AiTaskService

Responsible for:

* creating AI tasks;
* generating correlation identifiers;
* persisting immutable context snapshots;
* validating AI task lifecycle transitions;
* recording execution and failure metadata.

#### AIEngineClient

A future AI Engine client will be responsible for:

* submitting AI tasks through REST;
* handling communication timeouts;
* interpreting transport-level responses;
* reporting submission results to the workflow layer.

The workflow service will not contain HTTP client implementation details.

## Workflow Lifecycle

The initial analysis lifecycle is:

```text
PENDING
   │
   ▼
IN_PROGRESS
   │
   ├── COMPLETED
   └── FAILED
```

When the workflow begins, the analysis must be in a state that permits execution.

For V1, only a `PENDING` analysis may be started.

The initial preparation workflow is:

```text
Analysis PENDING
      │
      ▼
Start Workflow
      │
      ▼
Analysis IN_PROGRESS
      │
      ├── Generate Facts
      ├── Generate Observations
      ├── Build AnalysisContext
      └── Create AiTask
```

The exact point at which an analysis becomes `COMPLETED` will be refined as AI result processing is implemented.

Until then, the analysis remains `IN_PROGRESS` after the AI task is created.

A failure during workflow preparation changes the analysis status to `FAILED`.

## Workflow Result

The workflow should return an application-level result rather than exposing internal entities.

Conceptually:

```text
AnalysisWorkflowResult
├── analysisId
├── analysisStatus
├── factCount
├── observationCount
├── aiTaskId
├── aiTaskStatus
└── correlationId
```

The exact contract may evolve during implementation.

## Transaction Boundaries

The complete workflow will not execute inside one long database transaction.

Repository analysis, context construction, and future network communication may be long-running operations.

Keeping one transaction open throughout the complete workflow would:

* hold database resources unnecessarily;
* increase lock duration;
* make network failures part of a database transaction;
* complicate retries and recovery.

The workflow will instead use short transactional boundaries around individual state-changing operations.

Conceptually:

```text
Create / start Analysis        transaction

Persist Facts                  transaction

Persist Observations           transaction

Build AnalysisContext          read operation

Create AiTask                  transaction

Submit to AI Engine            external operation

Update submission state        transaction
```

Each specialized service remains responsible for its own transactional behavior.

The workflow service coordinates the sequence and handles failures between steps.

## Failure Handling

If a workflow preparation step fails before the AI task has been successfully prepared, the analysis must transition to `FAILED`.

Failure information should be logged and exposed through appropriate workflow or analysis metadata.

The workflow must not silently leave an analysis in `IN_PROGRESS` after an unrecoverable preparation failure.

When AI submission is introduced, failures will be distinguished between:

* deterministic analysis failure;
* context construction failure;
* AI task creation failure;
* AI submission failure;
* AI processing failure;
* AI result validation failure.

The exact retry strategy is outside the scope of this ADR.

## Idempotency

Workflow execution must not create duplicate deterministic results or duplicate AI tasks through accidental repeated calls.

For V1:

* a `PENDING` analysis may be started;
* an `IN_PROGRESS` analysis cannot be started again;
* a `COMPLETED` analysis cannot be restarted;
* a `FAILED` analysis requires a future explicit retry workflow.

The workflow service must reject unsupported repeated execution rather than attempting implicit recovery.

Future retry support must use explicit commands and lifecycle rules.

## Controller Boundary

REST controllers must remain thin.

A controller may trigger the workflow and return its result, but it must not directly coordinate:

* Analysis services;
* Fact services;
* Observation services;
* Context builders;
* AI task services;
* AI Engine clients.

The expected boundary is:

```text
Controller
    │
    ▼
AnalysisWorkflowService
    │
    ▼
Specialized domain and infrastructure services
```

## Initial Implementation Scope

The first implementation increment will include:

* `AnalysisWorkflowService`;
* workflow implementation;
* workflow result contract;
* validation of the initial analysis state;
* deterministic analysis invocation;
* `AnalysisContext` construction;
* `AiTask` creation;
* workflow unit tests;
* failure-state tests;
* prevention of duplicate starts.

The first increment will not include:

* a real AI Engine REST call;
* callback handling;
* automatic retry;
* cancellation;
* AI result conversion;
* proposal creation from callback results;
* knowledge promotion;
* distributed messaging.

## Alternatives Considered

### Orchestration inside the controller

Let the REST controller directly invoke all required domain services.

Rejected because:

* controllers would contain business workflow logic;
* the workflow would be difficult to reuse;
* testing would require controller-level orchestration tests;
* responsibility boundaries would become unclear.

### Orchestration inside AnalysisService

Add Facts, Observations, context construction, AI task creation, and AI submission to `AnalysisService`.

Rejected because:

* `AnalysisService` would become responsible for several domains;
* the Analysis aggregate would absorb application workflow concerns;
* the service would become increasingly difficult to maintain;
* domain and application responsibilities would be mixed.

### Event-driven orchestration

Publish domain events between every workflow step.

Rejected for V1 because:

* it introduces asynchronous distributed coordination prematurely;
* debugging and failure recovery become more complex;
* current processing volume does not justify this infrastructure;
* ADR-010 already selects asynchronous REST without a message broker for V1.

Internal domain events may be reconsidered later for secondary side effects.

### AI Engine orchestration

Let the Python AI Engine coordinate deterministic analysis, context retrieval, and proposal creation.

Rejected because:

* the Core would lose ownership of the business workflow;
* Python would require access to project knowledge;
* this would conflict with ADR-003, ADR-011, ADR-012, and ADR-013.

### One long transaction for the complete workflow

Rejected because:

* deterministic analysis may be long-running;
* future network communication cannot safely belong to a database transaction;
* locks and database resources would be held unnecessarily;
* recovery would be more difficult.

## Consequences

### Positive

* Establishes a clear application-level orchestration boundary.
* Keeps controllers thin.
* Preserves specialized service responsibilities.
* Makes the complete workflow visible and testable.
* Centralizes workflow ordering and state transitions.
* Prepares the Core for future AI Engine submission.
* Supports incremental implementation.
* Provides a clear location for future failure and retry coordination.

### Negative

* Introduces an additional application service.
* Requires explicit coordination between multiple domain services.
* Partial failures must be handled carefully.
* Short transactions require deliberate recovery behavior.
* Idempotency rules must be enforced.
* The workflow may grow and require decomposition as additional capabilities are introduced.
